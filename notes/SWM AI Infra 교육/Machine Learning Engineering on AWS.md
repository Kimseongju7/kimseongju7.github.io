## Module 0: 과정 소개

3일짜리 과정이고, 흐름은 "데이터 준비 → 모델 개발 → 운영·관리"로 이어진다.

- **Day 1 (데이터 기초)** — ML 개론과 도전 과제 분석, 데이터 처리와 특성 엔지니어링. (Module 1~4) 실습으로 Data Wrangler, EMR, Processing을 직접 써본다.
- **Day 2 (모델 개발)** — 모델링 방식 선택, 훈련·평가·튜닝, 배포 전략. (Module 5~8) 실습으로 하이퍼파라미터 최적화를 해본다.
- **Day 3 (운영·관리)** — 트래픽 이동, 보안, MLOps, 모니터링. (Module 9~11) 실습으로 SageMaker Pipelines·Model Registry, 데이터 드리프트 모니터링을 해본다.

전체를 관통하는 목표는 7가지로 정리할 수 있다: ML 기본기 이해, AWS 서비스로 데이터 처리·특성 엔지니어링, 문제 요구사항에 맞는 알고리즘 선택, 확장 가능한 ML 파이프라인 설계, 클라우드 ML 리소스 보안, CI/CD 자동화, 데이터 드리프트를 포함한 모니터링 전략 수립.

## Module 1: AWS의 ML 소개

기존 인공지능이 인간이 규칙을 넣어주는 규칙 기반 시스템(지능형 시스템)이었다면, ML(기계 학습)은 기계가 데이터를 통해 스스로 패턴과 규칙을 찾아내는 시스템이다. 기초 기술 → 머신 러닝 → 딥러닝 → 생성형 AI 순서로 기술이 발전해왔다고 이해하면 된다.

대표적인 활용 사례로는 개인화 추천(고객 속성 기반), 사기 탐지(거래 이상 패턴 감지), 예측 유지보수(센서 데이터 분석), 고객 이탈 방지(이탈 위험군 식별) 등이 있다.

### ML 워크플로우

1. **데이터 엔지니어링** — 수집, 탐색, 정제, 라벨링, 분할
2. **ML 파이프라인** — 모델 훈련, 평가, 테스트, 패키징
3. **배포 및 모니터링** — 실제 서비스 제공, 성능 추적
4. **피드백 기반 개선** — 다시 1번으로

학습 방식은 라벨된 데이터를 쓰는 **지도 학습**, 숨은 패턴을 스스로 찾는 **비지도 학습**, 환경과 상호작용하며 배우는 **강화 학습**으로 나뉜다. 이 워크플로를 굴리는 데는 데이터 과학자, 데이터 엔지니어, MLOps 엔지니어, 머신 러닝 엔지니어 같은 역할이 함께 필요하다.

### AWS SageMaker AI

"완전 관리형, End-to-end 머신 러닝 개발 플랫폼"이다. 파이프라인은 다음 순서로 이어진다.

> Data Wrangler → Feature Store → Notebooks → Canvas / Python SDK → Automatic Model Training → Endpoints → Model Monitor
>
> (데이터 준비 → 특성 저장 → Notebooks로 구축 → 모델 훈련 → 하이퍼파라미터 튜닝 → 프로덕션 배포 → 관리 및 모니터링)

훈련은 EC2 기반 Docker 컨테이너를 사용해 진행하며, 특성과 (레이블이 포함된) 훈련 데이터 세트를 입력으로 사용한다. 알고리즘은 크게 3가지 옵션을 제공한다.

- SageMaker 내장 알고리즘을 포함한 빌트인 Docker 이미지 — 지도학습 알고리즘(서포트 벡터 머신 등)
- TensorFlow, PyTorch 등의 프레임워크를 포함한 빌트인 Docker 이미지
- 커스텀 프레임워크 지원

### Responsible AI/ML

Responsible AI/ML이 필요한 이유는 생성형 AI 확산으로 가짜뉴스·정보 오용 위험이 커졌고, ChatGPT 같은 도구를 악용한 보안 공격도 실제로 존재하기 때문이다. AWS는 이를 8가지 요소로 정리한다.

1. **공정성** — 다양한 그룹에 미치는 영향 고려
2. **설명 가능성** — 모델이 왜 그런 결과를 냈는지 이해 가능해야 함
3. **투명성** — 시스템 정보를 명확히 공개
4. **개인정보보호/보안** — 데이터·모델을 안전하게 관리
5. **진실성/안정성** — 적대적 입력에도 정확성 유지
6. **거버넌스** — 책임 있는 관행을 정의하고 실행
7. **안전성** — 해로운 출력과 오용 방지
8. **제어 가능성** — 인간의 감독과 개입이 가능해야 함

## Module 2: ML 도전 과제 분석

머신러닝은 기계가 학습하여 패턴을 인식하고 예측을 수행하는 것이다. 다만 기존 프로그램(규칙 기반)으로 해결 가능하다면 그 방법을 쓰는 편이 낫다. ML 모델을 만드는 것은 예상보다 많은 비용과 시간이 든다.

### "이 문제, ML로 풀어도 될까?" 판단 기준 5가지

1. **패턴의 존재** — 단순 규칙으로는 못 푸는, 데이터 안에 숨은 패턴·관계가 있어야 한다.
2. **데이터 조건** — 충분한 양과 품질의 최신 데이터가 있어야 한다. 부족하거나 품질이 낮으면 애초에 부적합하다.
3. **지속적 개선 필요성** — 시장 트렌드, 고객 행동처럼 새 데이터가 계속 들어오고 그에 맞춰 자동으로 적응해야 하는 동적 환경에 잘 맞는다.
4. **명확한 성공 지표** — "고객 이탈율 10% 감소"처럼 정량화 가능한 목표가 있어야 검증할 수 있다.
5. **비교 분석** — 기존 솔루션과 벤치마크 비교로 ML이 정말 개선인지 확인해야 한다.

### SageMaker 기준 ML 개발 생명주기

- **입력** — 특성(features)과 레이블이 포함된 훈련 데이터
- **처리** — EC2 기반 Docker 컨테이너에서 훈련, 하이퍼파라미터 조정으로 성능 최적화
- **출력** — 학습된 가중치와 성능 지표를 포함한 모델

### ML 기법 분류

- **지도 학습**
  - 분류: 범주형 결과 예측 (SVM, 의사결정나무, XGBoost)
  - 회귀: 수치형 결과 예측 (선형회귀, KNN 회귀)
- **비지도 학습**
  - 클러스터링: 유사 데이터 그룹화 (K-Means, 계층적 클러스터링)
  - 연관: 변수 간 관계 발견 (Apriori, FP-Growth)
  - 차원 축소: 데이터 시각화·압축 (PCA)
- **심화 기법** — 딥러닝(비선형 복잡 문제), 컴퓨터 비전(픽셀 분석 기반 이미지 인식)

## Module 3: ML에 사용할 데이터 처리

데이터를 어디에서 가져오고, 어떻게 처리하고, 어떤 스토리지를 사용할 것인가에 대한 내용이다.

기계 학습용 데이터 소스는 다음과 같다.
- 데이터 레이크: 모든 형식의 데이터를 원본 그대로 저장
- 데이터 웨어하우스: 정형 데이터와 반정형 데이터를 정제·변환하여 저장
- 데이터베이스: 다양한 형식으로 애플리케이션이 이용하는 데이터를 저장

데이터 레이크와 데이터 웨어하우스는 애플리케이션이 직접 이용하는 저장소가 아니라 분석과 활용을 위한 데이터 저장소라는 점에서 데이터베이스와 다르다. 데이터 레이크는 호수처럼 온갖 형식의 데이터가 그대로 들어갈 수 있고, 데이터 웨어하우스는 csv, excel, json, xml 등 정제·변환된 데이터가 저장된다. 둘은 비슷해 보이지만 담기는 데이터의 속성이 다르다.

### 좋은 데이터의 조건

정확성(오류 없음), 완전성(누락 없음), 일관성(형식 통일), 관련성(문제 해결과 관련), 적시성(최신 상태), 신뢰성(출처 검증) — 이 6가지를 기준으로 데이터 품질을 점검한다. 구조로는 정형(규칙에 따라 저장), 비정형(영상·의료기록처럼 형태가 없는 데이터, 전체의 80~90%를 차지), 반정형(작성자 규칙을 따르되 완전히 정형은 아닌 데이터)으로 나뉜다.

데이터 훈련 전에는 막대그래프, 히스토그램, 밀도 플롯, 상자 플롯, [[상관관계행렬|상관관계 행렬]], [[열지도|히트맵]] 등으로 시각화해 데이터가 정상인지 확인할 수 있다. 상관관계행렬은 변수 간 선형 관계를 -1~1 값으로 보여주는데, 연관관계가 너무 밀접하면(다중공선성) 오버피팅·쏠림 위험이 있으므로 둘 중 하나는 특성에서 제외한다.

### SageMaker AI 모델 훈련 시 S3 사용 방법

- File mode: 훈련하기 전에 데이터셋을 연결된 Amazon EBS 또는 NVMe SSD 볼륨에 다운로드
- Fast File mode: 훈련 인스턴스의 파일 시스템 형태로 S3 스토리지를 연결해, 작성한 훈련 스크립트가 이를 읽어서 스트리밍
- Amazon EBS: EC2 인스턴스의 파일 시스템
- Amazon EFS: 여러 컴퓨팅 인스턴스가 공유 데이터셋, 훈련 코드, 사용자 지정 패키지, 모델 아티팩트에 동시 액세스해야 하는 다중 인스턴스 분산 훈련용 스토리지 (NAS와 유사)
- Amazon FSx for Lustre: Amazon S3 버킷과 연결해 import/export가 가능한 병렬 파일 시스템. 초당 수백만 건의 입출력 작업과 1밀리초 미만의 지연 시간을 제공

각 모드·스토리지는 데이터 크기, 훈련 속도, 비용 효율성에 따라 선택한다.

### 데이터 포맷

- 행 기반 데이터 저장: data ingestion, writing, ETL, 파이프라인에 유리. 전체 레코드를 빠르게 읽을 수 있지만 압축이 쉽지 않음 (csv, json, apache avro)
- 열 기반 데이터 저장: 모델 훈련, 분석, 특성 엔지니어링에 유리. 특정 데이터 유형의 압축이 빠름 (apache parquet, apache ORC, pandas)

| 포맷 | 특징 | 주 용도 |
|---|---|---|
| Avro | 스키마 기반 바이너리 직렬화 | Kafka 스트리밍, Hadoop/Spark |
| RecordIO | 청크 단위 바이너리 | MXNet, PaddlePaddle 훈련 |
| Parquet | 열 기반, 높은 압축률 | Spark 최적화, 빠른 열 읽기 |
| ORC | 열 기반, 업데이트 가능 | Hive 데이터웨어하우징 |
| JSON/JSONL | 계층적 구조, 유연함 | 비정형 데이터 (전처리 필요) |

## Module 4: 데이터 변환 및 특성 추출

전처리에 대한 내용이다. 데이터의 이상치, 형식 불일치, 중복값을 처리해야 할 뿐 아니라, 컴퓨터는 데이터 처리 시 문자열이 아닌 숫자로 처리하므로 이를 위한 인코딩이 필요하다. 또한 누락된 값은 대체(imputation)가 필요하다.

### 누락 데이터 유형

| 유형 | 특징 | 처리 방법 |
|---|---|---|
| 무작위 누락 (MAR) | 다른 변수와 연관성 있음 | 다른 변수로 대체·예측 |
| 완전 무작위 누락 (MCAR) | 어떤 변수와도 무관 | 해당 행·열 삭제 |
| 비무작위 누락 (MNAR) | 측정 불가한 변수값 자체 때문에 발생 | 복잡한 모델링, 전문가 지식, 다중 대체 |

[[Feature-Engineering|Feature Engineering]]은 이렇게 정리된 데이터를 원시 상태에서 변환·개선해 의미 있는 입력 변수를 만드는 작업으로, 모델의 성능을 향상시킨다.

### 범주형 변수와 인코딩

- 명목형: 각 범주 간 순위가 없는 변수 (성별, 국적, 혈액형)
- 순서형: 각 범주 간 순위가 있는 변수. 다만 범주 간 간격이 동일하다는 보장은 없음 (학력, 크기, 성적)

범주 종류에 따라 인코딩 방식이 달라진다.

#### 인코딩 방법
- 레이블 인코딩: 하나의 범주 값들을 0부터 연속적인 수치 데이터로 변환 (크기를 의미하지 않음). 단순히 1, 2, 3에 매핑하는 방식으로, 임의로 부여된 숫자가 크기나 순서로 비춰져 잘못된 학습을 유발할 위험이 있다.
- 원핫 인코딩: n개의 특성값을 n개의 비트 벡터로 표현하는 방식. 범주가 너무 많으면 새로운 열이 과도하게 생성되어 [[차원의-저주|차원의 저주]]가 발생한다. 숫자 간 크기 차이가 모델 학습에 영향을 미치는 선형회귀, 신경망 같은 알고리즘에 적합하다.
- [[서수 인코딩|순서형 인코딩]]: n개의 특성값을 0~n의 연속적인 숫자 데이터로 표현해 순서 정보를 유지
- 바이너리 인코딩: 범주형 변수를 이진 코드로 변환. 범주가 많을 때 메모리를 절약하지만 오버헤드가 생길 수 있음

의사결정트리, 랜덤 포레스트, CatBoost, 신경망처럼 인코딩이 아예 필요 없는 알고리즘도 있다.

### 수치형 변수 처리

#### 스케일링
- 정규화: 데이터 범위를 특정 구간(0~1 또는 -1~1)으로 제한. 이상치에 영향을 많이 받으며, 분포를 모르거나 정규분포가 아닐 때 사용 (KNN, 신경망, 거리 기반 알고리즘)
- 표준화: 데이터의 평균을 0, 표준편차를 1로 조정하여 표준정규분포를 따르게 함. 이상치 영향은 덜 받으며, 데이터가 정규분포를 따를 때 유용 (선형회귀, 로지스틱회귀, PCA)

#### 기타 변환 기법
- 비닝: 연속형 값을 이산형 구간으로 변환 (정보 손실 가능성 있음)
- 로그 변환: 연속형 값을 연속형으로 재변환 (정보 손실이 상대적으로 적음)

### 특성 선택: 주성분분석(PCA)

차원 축소를 위한 대표 기법으로, 절차는 이렇다.

1. 평균값이 원점이 되도록 데이터를 시프트
2. 원점을 지나는 직선에 데이터를 수직으로 내려 그 거리 제곱합(SS)이 최대가 되는 직선을 탐색
3. 그 직선을 첫 번째 주성분(PC1)으로 설정
4. 각 원래 변수가 PC1에 기여하는 정도(Loading Score) 계산
5. PC1과 직각을 이루는 다음 주성분을 순서대로 생성
6. Scree Plot으로 각 주성분이 설명하는 분산 비율을 시각화

## Module 5: 모델링 방식 선택

모델 개발은 "준비된 데이터 → 알고리즘 선택 → 훈련 → 테스트 결과 확인 → 출력 모델" 순서로 진행된다.

### 지도 학습 기본 제공 알고리즘

- **Linear Learner** — 선형 회귀·분류에 최적. 빠르고 확장 가능하지만 비선형 관계는 잘 못 잡는다.
- **XGBoost** — 하나의 모델을 훈련하면 오차가 발생하는데, 그 오차를 줄이는 방향으로 훈련을 계속 추가하는 부스팅 알고리즘이다. 회귀·이진/다중 분류·순위 문제에 두루 쓰이고 정확도가 높지만, 파라미터 튜닝이 필요하고 해석성은 낮은 편이다.
- **K-NN** — 거리 기반 비모수적 방법. 학습 과정이 단순하고 이해하기 쉽지만, 추론 비용이 높고 고차원 데이터에 약하다.
- **Factorization Machines** — 특성 간 상호작용을 모델링하며 희소 데이터셋에 강해 추천 시스템에 자주 쓰인다.

### 비지도 학습 알고리즘

| 알고리즘 | 목적 | 활용 분야 |
|---|---|---|
| Random Cut Forest (RCF) | 이상 탐지 | 시계열 데이터의 비정상 활동 감지 |
| IP Insights | IP 기반 이상 탐지 | 네트워크 보안 위협 탐지 |
| K-평균 | 클러스터링 | 고객 세분화, 데이터 그룹화 |
| LDA | 토픽 모델링 | 문서 주제 식별 |
| Object2Vec | 임베딩 생성 | 유사성 검색, 추천 |
| PCA | 차원 축소 | 특징 감소, 계산 속도 향상 |

### 텍스트/음성 & 딥러닝

- BlazingText: Word2vec, fastText 기반 워드 임베딩·텍스트 분류
- Sequence-to-sequence: 기계 번역, 텍스트 요약용 인코더-디코더 모델
- NTM(Neural Topic Model): LDA보다 고도화된 신경망 기반 토픽 모델링
- 프레임워크: MXNet(메모리 효율·분산 학습, AWS 지원), TensorFlow(커뮤니티·확장성·도구 우수)
- 시맨틱 세분화: 이미지 픽셀 단위로 클래스를 할당 (DeepLab 등)
- DeepAR: 시계열 예측용 LSTM 기반 RNN

### Amazon SageMaker Autopilot

자동 머신러닝(AutoML) 서비스로 회귀, 분류, 이미지/텍스트 분류, 시계열 예측을 지원한다.

- **Quick Mode**: 2~30분, 빠른 프로토타이핑, AutoGluon 기반
- **Standard Mode**: 2~5시간, 하이퍼파라미터 최적화(HPO)까지 수행해 높은 정확도 추구

### 모델 선택 시 고려사항

의료·금융처럼 고위험 영역에서는 모델이 왜 그렇게 판단했는지 설명 가능한 **해석 가능성**이 중요하고, 성능과 **비용** 사이의 균형도 함께 고려해야 한다.

## Module 6: 모델 훈련

### 모델의 구성 요소

ML 모델은 "실제 프로세스의 수학적 표현"이다. 알고리즘(학습 로직)과 파라미터(훈련 중 조정되는 가중치)로 구성되며, 훈련이란 예측값과 실제값의 차이를 최소화하도록 가중치를 조정해가는 과정이다.

### 데이터 분할과 적합도

데이터는 보통 훈련:검증:테스트 = 6:2:2 또는 8:1:1, 7:2:1 비율로 나눈다.

- **과대적합(overfitting)** — 훈련 데이터의 노이즈까지 학습해 새 데이터에 일반화하지 못함 → 데이터 다양성을 늘리거나 정규화로 완화
- **과소적합(underfitting)** — 기본 패턴조차 못 잡음 → 특성을 추가하거나 모델 복잡도를 높여 완화

### 손실 함수와 최적화

손실 함수는 훈련 데이터 대비 모델 예측 성능을 측정하는 지표다. 선형회귀에는 RMSE, 로지스틱 회귀에는 로그 가능성이 주로 쓰인다. 경사하강법으로 기울기를 계산하고, 학습률에 따라 가중치를 반복 업데이트하며 손실을 줄여나간다.

| 기법 | 사용 데이터 | 업데이트 빈도 | 속도 | 안정성 | 메모리 |
|---|---|---|---|---|---|
| 배치 경사하강법 | 전체 데이터 | 에포크당 1회 | 느림 | 매우 안정 | 높음 |
| 확률적 경사하강법(SGD) | 단일 샘플 | 데이터 수만큼 | 빠름 | 불안정(진동) | 낮음 |
| 미니배치 경사하강법 | 일부 샘플 | 미니배치 수만큼 | 균형 | 비교적 안정 | 적절 |

### Amazon SageMaker AI 훈련

훈련 방법은 세 갈래다.

1. **Canvas** — No-Code/Low-Code 환경
2. **Studio** — 완전관리형 코드 기반 대규모 훈련
3. **Pipelines** — ML 개발 자동화 오케스트레이션

SageMaker Python SDK의 주요 객체는 다음과 같다.

- **Estimators** — 훈련 작업을 캡슐화
- **Models** — 구축된 모델을 캡슐화
- **Predictors** — 실시간 추론 제공
- **Processors** — 데이터 처리 작업을 캡슐화
- **Transformers** — 배치 변환 작업을 캡슐화

예를 들어 XGBoost로 이진분류를 훈련한다면 "Estimator 생성 → 하이퍼파라미터 설정(objective, num_round, max_depth, eta, subsample) → fit() 호출" 흐름을 따른다. TensorFlow 기반 K-Means처럼 사용자 정의 스크립트를 쓸 때는 `train.py`에서 `SM_CHANNEL_TRAIN`, `SM_MODEL_DIR` 같은 환경변수로 입출력 경로를 지정한 뒤 TensorFlow Estimator로 실행한다.

### 컴퓨팅 인스턴스

- **CPU 인스턴스** (ml.m5, ml.c5) — 일반 ML 워크로드, 전처리, 초기 개발/테스트에 비용 효율적. 연산 집약 작업엔 느림
- **GPU 인스턴스**
  - P5 (NVIDIA H100) — 초대규모 분산 훈련, 최고 성능
  - G5 (NVIDIA A10G) — 성능과 비용의 균형
  - Trainium/Inferentia (AWS 자체 칩) — 특정 워크로드에서 비용 절감

### 모델 관리

**SageMaker Model Registry**가 모델 등록, 승인 워크플로, 자동화된 배포를 지원한다.

## Module 7: 모델 평가 및 튜닝

### 편향-분산 절충

- 과대적합: 낮은 편향, 높은 분산 (훈련 데이터에 과도하게 맞춰짐)
- 과소적합: 높은 편향, 낮은 분산 (일반화 부족)

### 분류 모델 평가 지표

[[혼돈행렬]]은 실제 정답과 예측을 긍정/부정 조합으로 나눠 표로 정리한 것이다. 긍정은 모델이 옳게 판단한 것, 부정은 아니라고 판단한 것을 뜻하며 네 칸으로 구성된다.

- **TP (True Positive)** — 정답을 정답이라 함
- **FP (False Positive)** — 오답을 정답이라 함
- **TN (True Negative)** — 오답을 오답이라 함
- **FN (False Negative)** — 정답을 오답이라 함

| 지표 | 정의 | 적용 사례 |
|---|---|---|
| 정확성(Accuracy) | (TP+TN) / (TP+TN+FP+FN) | 데이터가 균형 잡혀 있을 때 |
| 정밀도(Precision) | TP / (TP+FP) | 스팸 필터링처럼 오탐(오판)을 최소화해야 할 때 |
| 재현율(Recall) | TP / (TP+FN) | 암 진단처럼 실제 양성을 놓치는 것을 최소화해야 할 때 |
| F1 점수 | 정밀도와 재현율의 조화평균 | 정밀도·재현율 둘 다 중요할 때 |

### [[ROC]] 곡선과 AUC

모델은 확률값(0~1)을 출력하고, 임계값을 정해 분류를 결정한다. 이 임계값을 바꿔가며 X축은 FPR(False Positive Rate), Y축은 TPR(True Positive Rate)로 찍은 것이 ROC 곡선이다. 곡선이 좌상단에 가까울수록, 즉 곡선 아래 면적(AUC)이 클수록 모델 성능이 좋다.

### 훈련 시간 단축 기법

- **조기 중지** — 과대적합 방지와 컴퓨팅 시간 절약을 동시에
- **분산 훈련** — 다중 리소스 활용
- **SageMaker HyperPod** — 대규모 분산 학습 인프라 제공

**SageMaker Debugger**의 XGBoost 보고서에서는 레이블 분포, 손실-단계 추이, 특성 중요도(가중치·정보 증가), 혼동행렬, ROC 곡선, 잔차 분포까지 확인할 수 있다.

### 하이퍼파라미터 튜닝

탐색 전략은 그리드 검색(모든 조합 시도), 랜덤 검색(무작위 조합), 베이지안 최적화(이전 성능 기반 예측), 하이퍼밴드(효과적인 파라미터에 자원 집중) 네 가지가 대표적이다.

XGBoost의 주요 파라미터는 다음과 같다.

- `max_depth` — 트리 깊이 (과적합 제어)
- `eta` — 학습률 (낮을수록 느리지만 안정적)
- `gamma` — 분할 손실 임계값
- `min_child_weight` — 노드 최소 샘플 수
- `subsample` — 데이터 샘플링 비율
- `objective` — 목적함수 (binary:logistic, multi:softmax 등)

## Module 8: 모델 배포 전략

모델 배포는 훈련된 모델과 관련 리소스를 프로덕션 환경에 통합해 새 데이터에서 예측을 생성하는 과정이다. 핵심은 자동화되고 반복 가능한 배포 프레임워크를 만드는 것이다.

### 배포 전략 비교

| 전략 | 환경 구성 | 리소스 | 롤백 속도 | 적합한 상황 |
|---|---|---|---|---|
| [[롤링배포]] | 단일 환경에서 순차 교체 | 적음 | 순차적이라 상대적으로 느림 | 비용 제약이 있을 때 |
| [[블루그린배포]] | 완전히 분리된 두 환경 병행 운영 | 많음(일시적으로 이중) | 트래픽 전환만 하면 되어 즉각적 | 미션 크리티컬, 제로 다운타임이 필요할 때 |
| [[카나리아배포\|카나리아 배포]] | 일부 트래픽만 신버전에 우선 노출 | 중간 | 트래픽 비중을 0으로 되돌리면 즉시 | 위험을 점진적으로 검증하고 싶을 때 |

### Amazon SageMaker 추론 옵션

- **실시간 추론** — 즉각적인 응답이 필요할 때
- **[[서버리스]] 추론** — 인프라 관리 없이 자동 확장. 예측 불가능한 트래픽에 적합하지만 [[콜드 스타트]]가 발생할 수 있고 CPU만 지원
- **비동기 추론** — 대기열 기반, 대기 시간이 어느 정도 허용될 때, 대규모 요청·긴 처리시간에 적합
- **배치 추론** — 대량 데이터를 일괄 처리, 이벤트 기반 또는 예약 호출

### 엔드포인트 배포 방식

- **다중 모델 엔드포인트** — 공유 서비스 컨테이너에서 같은 ML 프레임워크를 쓰는 여러 모델을 함께 호스팅. `TargetModel` 파라미터로 특정 모델을 지정한다.
- **다중 컨테이너 엔드포인트** — 단일 엔드포인트에서 서로 다른 모델·프레임워크를 실행. 순차 파이프라인 또는 개별 호출 방식을 지원하며 `TargetContainerHostname`으로 컨테이너를 선택한다.

**Amazon SageMaker Inference Recommender**는 인스턴스 수·유형, 컨테이너 파라미터, 모델 최적화 수준, 최대 동시성, 메모리 크기 등을 평가해 최적의 배포 구성을 추천해준다.

비용과 안정성은 트레이드오프 관계이므로, 워크로드 특성에 맞춰 배포 전략과 추론 옵션을 함께 결정해야 한다.

## Module 9: AWS 기계 학습 리소스 보호 (보안)

ML 솔루션을 보호하기 위한 핵심 영역은 접근 제어, 네트워크 구성, CI/CD 파이프라인 보안 세 가지다.

- **IAM(Identity and Access Management)** — AWS 계정의 인증과 권한 부여를 담당한다. ML 솔루션 안에서 누가 어떤 리소스에 접근할 수 있는지 관리하는 기초가 된다.
- **VPC 기반 네트워크 격리** — SageMaker 도메인을 VPC Only Mode로 설정하면, 모든 트래픽이 지정한 VPC와 서브넷을 통해서만 흐르도록 강제할 수 있다. 기본값으로 열려 있던 SageMaker Studio 접근 경로를 안전한 네트워크 구조 안으로 좁히는 셈이다.
- **방화벽을 통한 트래픽 필터링** — VPC 환경 안에서 지원되는 방화벽으로 인바운드·아웃바운드 트래픽을 제어한다.

정리하면 3층 보안 구조로 볼 수 있다.

1. **접근 제어** — IAM으로 사용자·서비스 인증 및 권한 관리
2. **네트워크 격리** — VPC·서브넷으로 트래픽 범위 제한
3. **파이프라인 보안** — CI/CD 프로세스 자체와 그 기반 인프라 단계의 보안 확보

## Module 10: MLOps와 CI/CD 자동화

### MLOps란

"머신 러닝(ML) 모델을 구축, 배포 및 유지 관리하는 것을 자동화하고 간소화하는 일련의 사례"다. 머신러닝, DevOps, IT 운영을 하나로 통합하며 여러 역할 간의 협업을 강조한다.

핵심 구성요소는 다음과 같다.

- **데이터 관리** — 학습 데이터를 체계적으로 버전 관리하고, 전처리·특성 엔지니어링을 자동화
- **모델 학습/서빙** — 자동 훈련·평가·프로덕션 배포와 효율적인 서빙
- **CI/CD** — 코드 수정만으로 전체 파이프라인이 자동 실행되도록 구성
- **모니터링/재학습** — 지속적인 성능 모니터링 및 성능 저하 감지 시 자동 재학습 트리거
- **컨테이너화/오케스트레이션** — 모델과 의존성을 컨테이너로 패키징하고 쿠버네티스 등으로 운영
- **거버넌스** — 모델 투명성 강화, 규정 준수, 수명 주기 변경 사항 관리

### CI/CD 파이프라인의 자동화 테스트

- **단위 테스트** — AWS CodeBuild에서 pytest로 데이터 전처리 함수(결측값 처리, 특성 스케일링)의 정확성을 검증하고, JUnit XML 리포트와 COBERTURA 커버리지 리포트를 CodeBuild Reports에서 추적한다.
- **성능 테스트** — SageMaker Inference Recommender의 Load Test가 perfsize 부하 생성기로 다양한 구성의 엔드포인트 성능을 측정한다.
- **A/B 테스트** — SageMaker Endpoint의 다중 variant 구성으로 트래픽 가중치를 조정하며 피드백 기반으로 검증한다.

### Amazon SageMaker Pipeline

ML 워크플로 오케스트레이션 서비스로, 방향성 비순환 그래프(DAG) 형태로 단계를 정의한다. 여러 단계를 병렬 실행할 수도 있다.

- **Processing** — 결측값 채우기, 정규화, 데이터 분할 같은 전처리 스크립트 실행
- **Training** — 하이퍼파라미터 구성 및 모델 훈련
- **Condition** — MSE 같은 성능 지표가 기준치를 충족하는지 검증해 다음 단계 진행 여부 결정
- **RegisterModel** — 모델을 버전이 지정된 모델 패키지 그룹으로 Model Registry에 등록
- **CreateModel** — 배치 추론용 모델 생성
- **Transform** — 지정된 데이터셋에 대한 모델 예측 생성

### SageMaker Project

데이터 과학자의 개발 환경과 MLOps 엔지니어의 CI/CD 시스템을 통합 설정하는 엔터프라이즈 기능이다. 모델 빌드/배포용 샘플 코드가 담긴 다중 리포지토리, CodePipeline 또는 Jenkins 기반 자동 파이프라인, SageMaker Pipelines 실행, 모델 그룹 버전 관리와 엔드포인트 자동 배포까지 포함한다.

| 항목 | SageMaker Project | SageMaker Pipeline |
|---|---|---|
| 목적 | 전체 MLOps 수명 주기 관리·표준화 | ML 워크플로 특정 단계 자동화 |
| 범위 | 코드, 파이프라인, 모델 레지스트리, 엔드포인트 통합 관리 | 재사용 가능한 DAG 기반 단계 정의·실행 |
| 기능 | CI/CD 템플릿, 리소스 프로비저닝 제공 | 완전 관리형 서버리스 워크플로 오케스트레이션 |
| 관계 | 파이프라인을 포함해 함께 협력 | 독립적으로도 생성 가능하나, 프로젝트 안에 있으면 CI/CD를 부여받음 |

## Module 11: 모델 모니터링과 데이터 드리프트

모니터링 대상은 크게 세 영역이다. **모델 성능/데이터 품질**, **인프라**, **보안**. 이는 비즈니스 요구 충족, 인프라 성능 확인, 모델 성능 유지, 필요 시 재훈련, 규정 준수를 위해 필수적이다.

### 모델 드리프트(Model Drift)

"데이터 또는 입력 변수와 출력 변수 간의 관계 변화로 인해 머신 러닝 모델 성능이 저하"되는 현상으로, 잘못된 예측을 만들어낸다. 4가지 유형으로 나뉜다.

1. **데이터 품질 드리프트** — 입력 데이터의 통계적 속성(평균, 표준편차 등)이 훈련 데이터와 달라지는 현상. 프로덕션 데이터를 검증하고, 통계 지표를 계산해 훈련 데이터 기준과 비교하며 시간 경과에 따른 변화를 추적해 탐지한다.
2. **모델 품질 드리프트** — 모델 예측이 실제 정답 레이블([[Ground-Truth|Ground Truth]])과 달라져 평가 지표가 하락하는 현상. 성능 지표를 계산하고, 불확실한 예측에 임계값을 정해 플래그를 지정하며 지속 모니터링한다.
3. **편향 드리프트** — 훈련 데이터의 편향이나 실제 데이터 분포 변화로 편향이 생기는 현상. 추론 데이터의 편향 지표를 계산해 개발 시점 기준과 비교하고, 기준을 넘으면 완화 기법을 적용한다.
4. **특성 기여도 드리프트** — 개별 특성이 모델 예측에 미치는 영향도가 시간이 지나며 바뀌는 현상. SHAP 같은 설명 가능성 기법으로 특성 기여도 통계를 계산해 기준값과 비교한다. (SHAP 값이 양수면 예측을 높이는 방향, 음수면 낮추는 방향이며, 색상 진하기는 변수값 크기를 나타낸다.)

### Amazon SageMaker Model Monitor

프로덕션 환경에서 모델을 모니터링하는 포괄적인 솔루션으로, 데이터 품질·모델 성능·알림·드리프트 탐지를 제공한다. 동작 방식은 다음과 같다.

1. **모니터링 일정 설정** — 수집 대상, 빈도, 분석 방법, 보고 유형 결정
2. **데이터 캡처** — 엔드포인트를 통해 모델 입력과 예측값을 자동 수집
3. **기준선 생성** — 훈련 데이터로부터 통계·제약조건 생성 (모델 품질 모니터링은 [[Ground-Truth|Ground Truth]] 레이블도 추가 수집)
4. **기준 비교** — 실시간/배치 추론 결과를 기준 통계와 비교
5. **위반 신고 및 보고** — 임계값 초과 시 위반을 기록하고 [[Amazon CloudWatch]]로 모니터링

### 드리프트 감지 후 자동 재훈련 흐름

1. 사용자 정의 템플릿으로 SageMaker Project 생성
2. 전처리, 기준선 생성, 훈련을 포함한 Pipeline 구성
3. Model Registry에 등록하고 데이터 과학자가 검증·승인
4. 스테이징 엔드포인트 배포 후 프로덕션 배포
5. Model Monitor가 개념 편차(Concept Drift) 감지
6. [[Amazon CloudWatch]] 경보 → EventBridge 규칙 → 파이프라인 자동 재훈련 트리거

이 흐름을 매시간 실행되도록 스케줄링하거나 일정 기반으로 트리거해 모델을 지속적으로 갱신할 수 있다.
