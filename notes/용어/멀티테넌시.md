## 한 줄 정의
하나의 소프트웨어·인프라 인스턴스를 여러 고객(테넌트)이 함께 나눠 쓰되, 각자의 데이터와 설정은 서로 격리되도록 하는 아키텍처 방식.

## 자세히
멀티테넌시(multi-tenancy)에서 **테넌트(tenant)** 는 "세입자"라는 뜻으로, 서비스를 사용하는 하나의 고객 단위(보통 회사·조직)를 가리킨다. 한 채의 아파트 건물(하나의 시스템)에 여러 세대(여러 고객)가 입주해 살지만, 각 세대의 현관문·짐·생활은 서로 침범하지 않는 모습을 떠올리면 된다. 즉 물리적 자원은 공유하되 논리적으로는 분리되어, A 회사가 B 회사의 데이터를 보거나 건드릴 수 없어야 한다.

반대 개념은 **싱글테넌시(single-tenancy)** 로, 고객마다 전용 인스턴스·전용 서버를 따로 띄워 주는 방식이다. 싱글테넌시는 격리가 확실하고 커스터마이징이 자유롭지만, 고객 수만큼 자원과 운영 비용이 곱절로 든다. 멀티테넌시는 하나의 인스턴스를 여럿이 공유하기 때문에 **자원 효율과 비용, 운영·업데이트의 단순함**에서 크게 유리하다. 서버 한 번만 배포·패치하면 모든 고객에게 동시에 반영되므로, 대부분의 SaaS(Software as a Service) 제품이 이 모델을 택한다.

핵심 과제는 "어떻게 공유하면서도 안전하게 격리하느냐"이다. 데이터 격리 수준은 보통 세 가지로 나뉜다. ① **DB 분리** — 테넌트마다 별도의 데이터베이스를 두는 방식(격리 강함, 비용 높음), ② **스키마 분리** — DB는 공유하되 테넌트별 스키마를 나누는 중간 방식, ③ **공유 테이블 + 테넌트 ID** — 모든 테넌트가 같은 테이블을 쓰되 각 행에 `tenant_id` 를 붙여 구분하는 방식(자원 효율 최고, 격리는 애플리케이션 로직에 의존). 뒤로 갈수록 싸고 확장성이 좋지만, 쿼리에서 `tenant_id` 필터를 빠뜨리면 다른 고객 데이터가 새는 사고로 이어질 수 있어 격리 로직을 특히 엄격하게 다뤄야 한다.

또한 여러 테넌트가 자원을 공유하는 만큼, 한 테넌트가 과도한 부하를 일으켜 다른 테넌트까지 느려지는 **"시끄러운 이웃(noisy neighbor)"** 문제가 생길 수 있다. 그래서 테넌트별 사용량 제한(rate limiting)·자원 쿼터·성능 격리 같은 장치를 함께 두는 것이 보통이다.

## 예시
- **SaaS 제품**: Slack, Notion, Salesforce처럼 여러 회사가 같은 서비스에 각자 워크스페이스를 만들어 쓰는 형태가 전형적인 멀티테넌시다.
- **공유 테이블 + 테넌트 ID** 방식의 쿼리 — 모든 조회에 `tenant_id` 조건을 반드시 포함해야 한다.

  ```sql
  -- 현재 로그인한 테넌트의 주문만 조회
  SELECT * FROM orders
  WHERE tenant_id = :current_tenant_id
    AND status = 'paid';
  ```

  이 `WHERE tenant_id = ...` 를 실수로 빼면 전체 고객의 주문이 노출되므로, 보통 미들웨어/ORM 레벨에서 자동으로 테넌트 필터를 강제한다.
