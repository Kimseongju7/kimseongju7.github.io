## 한 줄 정의
[[서버리스]] 함수나 인스턴스가 한동안 쓰이지 않다가 다시 요청을 받을 때, 실행 환경을 새로 띄우느라 응답이 평소보다 느려지는 첫 실행 지연 현상.

## 자세히
콜드 스타트(cold start)는 AWS Lambda 같은 서버리스(Function as a Service) 환경에서 가장 자주 등장하는 개념이다. 서버리스는 코드를 상시 돌려두는 것이 아니라, 요청이 들어올 때만 컴퓨팅 자원을 [[프로비저닝]]해서 실행하고 끝나면 반납하는 [[온디맨드]] 방식으로 동작한다. 문제는 요청이 왔을 때 재사용할 수 있는 기존 실행 환경(컨테이너)이 없으면, 그제서야 새 컨테이너를 만들고 런타임을 초기화하고 코드를 로드하는 과정을 거쳐야 한다는 점이다. 이 초기화 과정 전체가 콜드 스타트이고, 그동안 사용자는 평소보다 훨씬 긴 응답 시간을 겪는다.

반대로 이미 초기화된 실행 환경이 남아 있어서 바로 코드 실행 단계로 넘어가는 경우를 **웜 스타트(warm start)** 라고 부른다. 클라우드 제공자는 한 번 쓰인 실행 환경을 일정 시간(보통 수 분~수십 분) 동안 유지해서, 짧은 간격으로 다시 요청이 오면 웜 스타트로 처리되게 한다. 하지만 트래픽이 뜸하거나, 동시 요청이 많아 새 인스턴스를 추가로 띄워야 하거나, 배포로 코드가 바뀌면 다시 콜드 스타트가 발생한다.

콜드 스타트의 지연 시간은 런타임과 패키지 크기에 크게 좌우된다. 예를 들어 Node.js·Python처럼 가벼운 런타임은 수백 ms 이내로 끝나는 경우가 많지만, JVM(Java)이나 .NET처럼 초기화 비용이 큰 런타임, 혹은 의존성이 무거운 패키지는 수 초까지 늘어질 수 있다. 이 때문에 응답 지연에 민감한 서비스(결제, 실시간 API 등)에서는 콜드 스타트가 실질적인 성능 리스크가 된다.

이를 줄이기 위한 대표적인 방법이 **프로비저닝된 동시성(Provisioned Concurrency)** 이다. 일정 개수의 실행 환경을 미리 초기화된 상태로 대기시켜 두는 기능으로, 요청이 오면 콜드 스타트 없이 바로 웜 상태로 처리한다. 다만 이는 대기 상태에서도 비용이 청구되므로, 온디맨드의 "쓴 만큼만 낸다"는 이점과는 트레이드오프 관계에 있다. 이 밖에도 함수 패키지 경량화, 불필요한 초기화 코드 지연 로딩, 언어/런타임 선택 등이 콜드 스타트를 줄이는 실무적 방법이다.

개념 자체는 서버리스에 국한되지 않는다. 캐시가 비어 있어 첫 조회가 느린 경우, [[Amazon-EC2-Auto-Scaling|Auto Scaling]]으로 새 인스턴스가 막 켜져 아직 워밍업이 안 된 경우 등도 넓은 의미의 콜드 스타트라고 부른다.

## 예시
- Lambda 함수를 며칠 만에 처음 호출하면 응답이 1~2초 걸리지만, 곧이어 다시 호출하면 수십 ms 만에 끝난다 — 첫 호출이 콜드 스타트, 이후가 웜 스타트.
- 트래픽 급증으로 Lambda가 동시 실행 인스턴스를 추가로 띄울 때, 새로 띄운 인스턴스들은 각각 콜드 스타트를 겪는다.
- Provisioned Concurrency를 5로 설정하면, 항상 5개의 실행 환경이 초기화된 채 대기하다가 요청을 즉시 웜 상태로 처리한다.
