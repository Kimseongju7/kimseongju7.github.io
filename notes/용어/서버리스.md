## 한 줄 정의
서버가 없다는 뜻이 아니라, 서버의 프로비저닝·확장·유지관리를 클라우드 제공자가 대신 맡아서 개발자는 서버 관리 없이 코드 실행에만 집중할 수 있는 클라우드 실행 모델.

## 자세히
서버리스(serverless)는 이름과 달리 서버가 실제로 없는 게 아니다. 서버는 여전히 존재하지만, 그 서버를 **프로비저닝하고, 패치하고, 스케일링하는 책임이 사용자에게서 클라우드 제공자로 완전히 넘어간** 상태를 가리킨다. EC2처럼 인스턴스를 직접 띄워두고 운영체제·용량·오토스케일링을 신경 써야 하는 방식과 달리, 서버리스에서는 코드(함수)만 올리면 요청이 들어올 때 제공자가 알아서 실행 환경을 켜고, 처리가 끝나면 끈다. [[온디맨드]]·[[프로비저닝]] 개념이 극단으로 밀어붙여진 형태라고 볼 수 있다.

가장 대표적인 예가 **AWS Lambda** 같은 FaaS(Function as a Service)다. 요청이 없을 때는 아무 자원도 켜져 있지 않고 비용도 0에 가깝다. 요청이 들어오면 그 순간 실행 환경이 생성돼 함수를 돌리고, 끝나면 사라진다. 그래서 과금 방식도 "서버를 켜둔 시간"이 아니라 **실제 코드가 실행된 시간과 호출 횟수**를 기준으로 한다. 이 점이 EC2 [[온디맨드]] 인스턴스와 다른 지점인데, EC2는 인스턴스를 켜두는 동안은 요청이 없어도 요금이 나가지만, 서버리스 함수는 아예 호출이 없으면 비용이 발생하지 않는다.

서버리스의 장점은 명확하다. 인프라 관리 부담이 사라지니 개발자는 비즈니스 로직에만 집중할 수 있고, 트래픽에 맞춰 자동으로 확장·축소되니 [[Amazon-EC2-Auto-Scaling|오토스케일링]]을 직접 설계할 필요가 없다. 트래픽이 뜸한 서비스나 이벤트 기반 작업(이미지 업로드 후 썸네일 생성, 주기적 배치 작업 등)에 특히 잘 맞는다. 다만 공짜는 아니어서, 함수가 오래 유휴 상태였다가 처음 호출될 때 실행 환경을 새로 켜는 지연([[콜드 스타트]]), 실행 시간·메모리 제한 같은 플랫폼 제약, 여러 함수로 쪼개질수록 커지는 관측·디버깅의 어려움이 트레이드오프로 따라온다. 이런 특성 때문에 서버리스는 흔히 [[마이크로서비스 아키텍처]]와 함께 묶여, 하나의 작은 기능 단위를 함수 하나로 배포하는 식으로 쓰인다.

## 예시
- **AWS Lambda**: S3에 이미지가 업로드되는 이벤트를 트리거로 함수가 실행돼 썸네일을 만들고 종료 — 그 사이에만 과금.
- **서버리스 vs EC2 온디맨드 비교**:
  - 서버리스(Lambda) — 서버 관리 없음, 호출·실행시간 단위 과금, 콜드 스타트 존재
  - EC2 온디맨드 — 인스턴스를 직접 켜둠, 켜져 있는 시간만큼 과금, 관리 부담 있음
- **API Gateway + Lambda + DynamoDB**: 서버 한 대 없이 REST API 백엔드 전체를 구성하는 전형적인 서버리스 스택.
