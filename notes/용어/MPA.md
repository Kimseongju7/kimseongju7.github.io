## 한 줄 정의
MPA(Multi Page Application, 다중 페이지 애플리케이션)는 링크를 누를 때마다 브라우저가 서버에 **새 HTML 문서를 요청**하고 화면 전체를 다시 그리는, 전통적인 방식의 웹 애플리케이션이다.

## 자세히
MPA는 웹의 가장 오래된 기본 구조다. 사이트가 여러 개의 HTML 페이지(`/`, `/about`, `/posts/1` …)로 이루어져 있고, 사용자가 링크를 클릭하거나 폼을 제출할 때마다 브라우저가 서버에 해당 주소의 HTML을 요청한다. 서버는 필요한 데이터를 채워 **완성된 HTML 한 장**을 응답하고, 브라우저는 받은 문서로 화면을 통째로 다시 그린다. 그래서 페이지를 옮길 때마다 화면이 잠깐 하얗게 깜빡이는 새로고침이 생긴다. [[SPA]]가 "하나의 HTML 안에서 JS로 화면만 바꿔치기"하는 것과 정반대다.

동작 방식이 단순한 만큼 장점이 분명하다. 첫째, **첫 화면이 빠르고 SEO에 강하다**. 서버가 내용이 다 담긴 HTML을 바로 내려주므로, 방문자도 검색엔진 크롤러도 도착하자마자 완성된 콘텐츠를 본다([[CSR]]의 약점을 처음부터 겪지 않는다). 둘째, 브라우저의 **뒤로가기·주소창·북마크가 그냥 동작한다**. 페이지 단위로 URL이 나뉘어 있으니 라우팅을 따로 흉내 낼 필요가 없다. 셋째, 프론트엔드에 무거운 JS 프레임워크가 없어도 되므로 구조가 가볍고 이해하기 쉽다.

단점은 **페이지 전환마다 서버 왕복과 전체 새로고침이 일어난다**는 점이다. 이미 떠 있는 헤더·사이드바처럼 바뀌지 않는 부분까지 매번 다시 받아 다시 그리니, 상호작용이 잦은 앱에서는 느리고 끊기는 느낌을 준다. 그래서 Gmail·Notion 같은 "한 화면 안에서 계속 머무는" 서비스는 MPA 대신 SPA로 갔다.

정리하면 MPA와 SPA는 대립하는 두 극단이고, 현대 웹은 그 사이를 오간다. MPA의 강점(빠른 첫 화면·SEO)을 SPA에 이식한 게 [[SSR]](서버 사이드 렌더링)이다 — 서버가 완성된 HTML을 먼저 내려주되(=MPA처럼), 그 뒤 JS가 붙어 이후 전환은 SPA처럼 매끄럽게 만든다. 이 블로그가 쓰는 [[Jekyll·Chirpy]]는 빌드 시점에 HTML을 미리 만들어 두는 SSG지만, 방문자 입장에서는 페이지마다 별도 HTML을 받는 **MPA로 동작**한다.

## 예시
- **대표 사례**: 위키백과, 뉴스 사이트, 쇼핑몰의 상품 목록↔상세, 전통적인 게시판, 그리고 이 [[Jekyll·Chirpy]] 블로그 — 링크를 누르면 새 HTML 페이지로 넘어간다.
- **동작 흐름**: 사용자가 목록에서 글 제목을 클릭하면 → 브라우저가 서버에 `GET /posts/1` 요청 → 서버가 글 내용이 채워진 HTML을 응답 → 브라우저가 화면 전체를 새 페이지로 교체(주소창도 실제로 바뀜, 새로고침 발생).

```
MPA 흐름:
브라우저 → 서버   : GET /posts/1
서버              : DB 조회 → HTML 완성
서버   → 브라우저 : 내용이 다 채워진 완성된 HTML (전체 새로 그림)

SPA 흐름 (대비):
브라우저          : 링크 클릭 → 서버에 HTML 요청 안 함
브라우저 → 서버   : GET /api/posts/1 (JSON만)
브라우저          : 받은 데이터로 화면 일부만 교체 (새로고침 없음)
```

## 관련
- [[SPA]] — MPA와 정반대인 단일 페이지 방식, 이 노트의 짝
- [[CSR]] — MPA(서버가 HTML 완성)와 대비되는 클라이언트 렌더링
- [[SSR]] — MPA의 첫 화면·SEO 강점을 SPA에 결합한 방식
- [[SEO]] — MPA가 유리한 검색엔진 최적화
- [[Frontend 공부하기]] — 렌더링 방식 전반을 정리한 노트
- [[Jekyll·Chirpy]] — 이 블로그가 쓰는 SSG, 방문자에겐 MPA로 동작
- [[DOM]] — MPA가 페이지 이동마다 통째로 다시 그리는 문서 구조
- [[메타프레임워크]] — MPA와 SPA의 장단점을 페이지 단위로 섞어 쓸 수 있게 해주는 도구
