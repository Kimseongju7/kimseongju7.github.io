## 한 줄 정의
누가(주체) 무엇에(리소스) 어떤 작업을 할 수 있는지/없는지를 규칙으로 적어 둔 문서로, AWS IAM에서는 **JSON 형식의 권한 규칙 묶음**을 가리킨다.

## 자세히
넓게 보면 "정책 문서(policy document)"는 조직·시스템이 지켜야 할 규칙을 글로 정리한 것 전반을 뜻하지만, 클라우드/AWS 맥락에서는 거의 항상 **IAM 권한 정책을 담은 JSON 문서**를 의미한다. 사람이 말로 "이 사람은 S3만 읽을 수 있어"라고 하는 대신, 기계가 판단할 수 있도록 규칙을 구조화해 적어 놓은 것이다.

정책 문서는 몇 개의 **Statement(문장)**로 이루어지고, 각 문장은 보통 다음 요소를 가진다.
- **Effect** — 허용(`Allow`) 또는 거부(`Deny`).
- **Action** — 대상 작업 (예: `s3:GetObject`, `ec2:StartInstances`).
- **Resource** — 작업이 적용될 대상 (예: 특정 S3 버킷 ARN).
- **Condition**(선택) — 조건 (예: 특정 IP·시간·MFA 인증일 때만).

권한은 **최소 권한 원칙**에 따라 필요한 것만 부여하는 게 원칙이고, 기본은 "모두 거부"라서 명시적으로 `Allow` 한 것만 허용된다. 단, 어느 한 곳에서라도 `Deny`가 걸리면 다른 곳의 `Allow`를 이기고 최종적으로 거부된다(명시적 거부 우선).

정책은 붙는 위치에 따라 두 종류로 나뉜다. **아이덴티티 기반 정책**은 IAM 사용자·그룹·역할에 붙어 "이 주체가 무엇을 할 수 있는가"를 정하고, **리소스 기반 정책**은 S3 버킷·API 게이트웨이 같은 리소스에 직접 붙어 "누가 이 리소스에 접근할 수 있는가"를 정한다. 실무에서는 역할(개발자·운영자·데이터 사이언티스트 등)별로 정책 문서를 나눠 관리한다.

## 예시
S3 버킷 하나를 읽기만 허용하는 최소 정책 문서:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}
```

`Version`은 정책 언어 버전(날짜 형식의 고정값)이고, 위 문서를 IAM 사용자에게 붙이면 그 사용자는 `my-bucket`에서 객체를 읽고 목록을 볼 수만 있다.
