## 한 줄 정의
AI에게 직접 한 줄씩 명령하는 대신, **AI에게 자동으로 명령을 내리는 시스템(루프)을 설계**하는 것 — 즉 사람이 "엔진"에서 "엔진 설계자"로 역할을 옮기는 작업.

## 자세히
지금까지는 사람이 키보드 앞에 앉아 AI한테 한 줄씩 시켰다면, Loop Engineering에서는 **사람이 루프 밖으로 빠지고 루프가 알아서 AI를 굴린다.** 핵심 문장은 *"AI에게 명령하지 말고, 명령하는 시스템을 설계하라."* 이 용어는 2026년 6월 Peter Steinberger, Boris Cherny(Anthropic Claude Code 리드), Addy Osmani(Google Chrome 엔지니어)가 거의 동시에 같은 얘기를 꺼냈고, Osmani가 "Loop Engineering"이라 이름 붙였다.

### 4층 스택 — 루프는 맨 윗층
"XX 엔지니어링" 용어들은 대체가 아니라 **쌓이는 구조**다. 위로 갈수록 다루는 단위가 커진다.

| 층 | 다루는 것 | 핵심 질문 |
|---|---|---|
| **Prompt** | 한 문장 잘 쓰기 | 모델한테 뭐라고 말할까 |
| **Context** | 지금 창에 뭘 넣을까 | 무엇을 넣고/요약하고/비울까 |
| **Harness** | 한 번의 실행 무장 | 어떤 도구·행동, 뭘 '완료'로 볼까 |
| **Loop** | **스스로 반복 실행** | 어떻게 알아서 계속 돌게 할까 |

루프는 하네스가 남겨둔 *"사람이 버튼 누르길 기다리는 부분"*을 자동화한다.

### 한 턴을 이루는 5가지 동작 (Five Moves)
하나라도 빠지면 루프가 안 돌거나 제자리만 돈다.

1. **Discovery (발견)** — 이번에 뭘 할지 *스스로* 찾는다 (실패한 CI, 열린 이슈, 최근 커밋). 루프 품질의 천장을 결정.
2. **Handoff (위임)** — 찾은 일을 작업 에이전트에 넘긴다. 작업마다 **격리된 git worktree**로 충돌 방지.
3. **Verification (검증)** — *다른* 에이전트가 결과를 채점. **"안 돼(no)"라고 말할 수 있는 단계.** 가장 빼먹기 쉽고 가장 빼면 안 됨.
4. **Persistence (저장)** — 결과를 대화창 밖(파일·보드)에 남긴다. PR·티켓·상태 파일.
5. **Scheduling (스케줄링)** — 타이머로 반복. 이게 있어야 "한 번 실행"이 진짜 "루프"가 됨.

이를 실현하는 **6가지 부품(Parts)**: 자동화(Automations), 워크트리(Worktrees), 스킬(Skills), 커넥터(Connectors=MCP), 서브에이전트(Sub-agents), 메모리(Memory).

### 가장 중요한 통찰: 생성자 vs 평가자 분리
**AI에게 자기 작품을 채점시키면 거의 항상 칭찬한다.** 자기가 쓴 코드를 보면 결과물이 아니라 "내가 왜 이렇게 썼는지에 대한 변명"이 보이기 때문. 루프 안에서 반복되면 매 턴 자기한테 고개만 끄덕이며 품질에서 멀어진다.

해결책은 생성자를 더 비판적으로 만드는 게 아니라 **의심 많은 별도의 평가자(Evaluator)를 두는 것:**
- **Generator(생성자)**: 코드를 쓴다.
- **Evaluator(평가자)**: *다른 모델*, "이 코드는 망가졌다"고 가정하고 시작. 칭찬 금지.
- 평가자는 **읽기만 하지 말고 직접 행동** — Playwright MCP로 페이지 열고, 버튼 누르고, 스크린샷 찍어 "실제로 작동"을 판단.
- (GAN에서 빌려온 아이디어: 하나는 만들고, 하나는 흠을 잡는다.)

Claude Code의 `/goal`이 이걸 제품화한 것: 조건 충족될 때까지 돌고, **매 턴 끝에 작은 다른 모델이 "조건 충족됐나?" 판정** (은행의 maker-checker 원칙 적용).

### 루프가 망가지는 5가지 방식
각 실패 = 5가지 동작 중 하나를 빼먹은 것.
- **Nodding loop** — 검증 생략. 자기 코드에 자기가 합격 도장. *가장 흔함.*
- **Amnesiac loop** — 저장 생략. 매일 아침 똑같은 데서 다시 시작.
- **Manual loop** — 스케줄링 생략. 시연한 날만 돌고 끝.
- **Blind loop** — 발견 생략. 사람이 여전히 일거리를 줌.
- **Tangled loop** — 위임 생략. 여러 에이전트가 같은 폴더 건드려 충돌.

### 조용히 쌓이는 4가지 비용
서로를 강화하며 한꺼번에 청구된다.
1. **Verification debt (검증 부채)** — "돌아간다"와 "옳다" 사이의 미검증 출력.
2. **Comprehension rot (이해력 부패)** — 내가 안 쓴 코드가 늘수록 머릿속 지도와 실제 코드의 격차.
3. **Cognitive surrender (인지적 항복)** — "이제 굳이 안 본다." 판단 외주화.
4. **Token blowout (토큰 폭발)** — 버그 하나가 밤새 헛돌며 청구서만 키움.

### 결론
> **루프는 충실한 곱셈 기호다. 무엇을 곱하든 그건 사람이다.**

루프는 "생성"을 거의 공짜로 만들고, **희소해지는 것은 "판단"**이다. 같은 루프라도 만드는 사람에 따라 정반대 결과 — 이해를 가져오면 6개월 뒤 더 강해지고, 게으름을 가져오면 자기가 못 읽는 기계의 문지기로 전락한다. 코드 90%가 똑같아도 **체크포인트 1~2개 차이**가 누가 통제권을 쥐는지를 가른다.

## 예시
실제 사례:
- **개인** — Osmani의 아침 트리아지 루프: 매일 아침 자동 실행, CI/이슈/커밋 읽고, 고칠 거리마다 worktree 열고, 한 에이전트가 수정·다른 에이전트가 리뷰, PR 자동 생성. 못 다루는 건 inbox로.
- **기업** — Stripe "Minions": 주당 **1,300개 PR을 손으로 한 줄도 안 쓰고 자동 병합.** 핵심 교훈은 *강한 모델이 아니라 제약(constraints)의 품질이 신뢰성을 만든다.* 오픈소스 도구 Goose의 포크. PR은 여전히 사람이 리뷰 — 사람이 "작성→검토"로 책상만 옮긴 것.

첫 루프 만드는 순서: ① `/loop`로 타이머 실행 → ② CI·이슈 읽는 트리아지 스킬 → ③ 상태 파일 추가 → ④ `/goal` 평가자 추가(가장 중요·가장 빼먹기 쉬움) → ⑤ worktree로 병렬화. **작게 시작하되 "no라고 말하는 검증"과 "사람 리뷰 지점"은 처음부터 넣어라.**
