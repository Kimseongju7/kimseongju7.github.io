## 한 줄 정의
서버(인스턴스) 전체를 한 번에 교체하지 않고, 일부씩 순차적으로 새 버전으로 교체해가는 배포 전략이다.

## 자세히
여러 대의 서버가 로드밸런서 뒤에서 같은 서비스를 처리하고 있다고 하면, 롤링배포는 이 서버들을 한꺼번에 내리지 않고 몇 대씩 그룹으로 나눠 순서대로 새 버전을 올린다. 예를 들어 서버가 10대라면 먼저 2대를 트래픽에서 빼고 새 버전을 배포한 뒤 헬스체크를 통과하면 다시 트래픽에 투입하고, 다음 2대로 넘어가는 식이다. 이 과정을 전체 서버가 새 버전으로 바뀔 때까지 반복한다.

가장 큰 장점은 서비스 중단(다운타임) 없이 배포할 수 있다는 점이다. 항상 일정 비율 이상의 서버가 살아서 트래픽을 처리하기 때문에 사용자 입장에서는 배포가 진행 중인지 알기 어렵다. 또한 블루/그린 배포처럼 전체 인프라를 두 배로 띄울 필요가 없어 비용 효율도 좋다.

단점도 있다. 배포 도중에는 구버전과 신버전이 동시에 트래픽을 받는 구간이 생기기 때문에, API 응답 스키마나 DB 마이그레이션처럼 버전 간 호환성이 깨지는 변경이 있으면 문제가 생길 수 있다(하위 호환을 반드시 고려해야 한다). 그리고 문제가 생겼을 때 롤백도 순차적으로 진행되기 때문에, 블루/그린 배포처럼 트래픽을 즉시 이전 버전으로 전환하는 방식보다 되돌리는 속도가 느리다.

Kubernetes의 Deployment 오브젝트는 기본 배포 전략(`RollingUpdate`)으로 롤링배포를 사용하며, `maxSurge`(한 번에 추가로 띄울 수 있는 최대 인스턴스 수)와 `maxUnavailable`(한 번에 내릴 수 있는 최대 인스턴스 수) 값으로 배포 속도와 안정성 사이의 균형을 조절한다. AWS의 EC2 Auto Scaling Group이나 Elastic Beanstalk에도 비슷한 롤링배포 옵션이 있다.

## 예시
```yaml
# Kubernetes Deployment의 롤링배포 설정 예시
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 한 번에 1대까지 더 띄울 수 있음
      maxUnavailable: 0  # 배포 중에도 트래픽 처리 가능한 대수를 유지
```
