2026년 7월 10일, 정문창 멘토와 팀원들이 진행한 멘토링 세션이다. 개발 중인 문서 버전 관리 서비스(글로우)의 **분기(Branch) 기능 UX와 정책 방향**을 집중적으로 논의했다. 핵심 쟁점은 분기 기능의 직관성 부재, 타겟 사용자 재정의, 시맨틱 Diff의 범위, 그리고 정책 문서화의 필요성으로 이어졌다.

> 한 줄 요약: 분기 기능의 UX 직관성 부재와 정책 미정의 문제를 진단하고, **타깃 유저 재정립**과 **정책 문서화**를 최우선 과제로 정한 세션.

## 분기(Branch) 기능의 직관성 문제

분기 기능 자체는 의미 있는 시도로 평가되었으나, 이를 **사용자에게 직관적으로 설명하고 UX로 구현하는 데 어려움**이 있다는 점이 핵심 문제로 지적되었다. 서비스명 '글허브'가 GitHub에서 영감을 받은 만큼 Git 사용 개발자에게는 분기 개념이 자연스럽지만, 일반 사용자에게는 혼란을 줄 수 있다.

- 분기를 통해 사용자가 **실질적으로 얻는 가치**가 무엇인지 먼저 정의해야 한다. 분기 개수 설정보다 이점 정의가 우선이며, 그 이점에 맞춰 UX와 운영 정책을 설계해야 한다.
- **Merge 미지원 문제**: 병합 기능을 제공할 계획이 없다면 분기의 존재 의미가 사용자에게 불명확해진다. 피그마처럼 "사용자가 알아서 합치는" 방식도 검토되었으나, 이는 병합 책임을 사용자에게 전가하는 것으로 평가되었다.
- 고유 포맷이나 텍스트 파일 형식이 없어 데이터 병합(Merge) 구현을 확답하기 어렵고, 피그마 환경에서도 병합 지원은 제한적이라는 점을 고려해야 한다.
- 분기 개념이 오히려 복잡도를 높이고 사용자에게 판단 부담을 준다는 우려가 반복되었다. 멘토는 **"분기를 씀으로써 파일로 관리할 때의 불편함이 실제로 해소되는가"**를 근본적으로 재검토해야 한다고 강조했다.
- 파일 하나에서 여러 브랜치를 만들면 최종 버전을 식별하기 어렵고 브랜치 간 차이도 구분하기 어려워, 기존 v1·v2 순차 관리보다 오히려 버전 관리가 복잡해질 수 있다.
- 분기 기능을 없앨 경우 이전 버전과의 비교만 가능해지는 한계가 있어, 기능 유지와 사용자 경험 사이의 고민이 필요하다.

## 버전·분기 생성 정책과 네이밍

- 현재 구조에서는 **v3에서는 분기 생성이 불가능하고 v4에서만 지원**된다. v1에는 이미 분기가 존재하므로 v2에 분기를 생성하는 방향으로 진행하기로 했으나, 버전별 분기 생성 개념이 혼란스럽다는 의견이 나왔다.
- 최신 버전이 아닌 **과거 버전에서 분기를 생성하는 시나리오**는 일반적인 사용 패턴과 거리가 있다. 문서 편집은 주로 새 내용을 추가·삭제하는 과정이며, 과거 데이터는 삭제하지 않고 포함한 채 변경하는 경우가 일반적이다. 과거 시점에서 분기를 만들어 현재 문서에 병합하는 사례는 확률적으로 매우 낮다.
- **최종 버전에서 분기 생성이 불가능**하면, 분기를 만들기 위해 불필요하게 새 버전을 생성해야 하는 번거로움이 발생한다. 과거보다 최신 버전을 기준으로 분기를 생성하는 것이 일반적인 사용 환경에 더 적합하다.
- 다중 사용자 편집 시 버전 분기가 중복 생성될 수 있다. 다수 동시 편집을 대비해 **분기 개수를 현재 1개에서 2개 이상으로 확대**하되, 단일 분기만 허용하고 머지 책임을 사용자에게 맡기는 방안도 검토되었다. 두 명으로 인원을 제한하지 않는 방향으로 구현하기로 했다.
- **네이밍**: 사용자가 직접 'v버전' 규칙을 설정하는 대신 기본값을 유지하고 커뮤니티 추천 방식을 활용한다. 분기가 여러 개면 현재 네이밍 규칙을 유지하기 어려우므로 구성 자체를 공유하는 방식을 검토한다. 기본 버전 이름(v2, v3 등)을 자동 부여하되 사용자가 수정할 수 있게 하고, 커밋 메시지를 입력하지 않으면 제목으로 대체한다. 분기 이름 뒤 식별자는 의미 없는 문자열로 대체 가능하므로 현 단계에서 명명 규칙을 깊이 고민할 필요는 없다.
- 분기 제한 정책은 구현상의 제약이 반영된 결과일 수 있으므로, **구현 방식보다 기능의 정의와 정책 수립을 먼저** 사고해야 한다. 구현을 동시에 고려하면 생각이 혼재된다.

## 업로드·모달·하이라이트 UX

- **모달 종속성**: 특정 화면(페이지) 내에서 열리는 업로드 모달이 폴더·버전을 동시에 변경할 수 있게 되어 있어, 현재 화면에 종속된 맥락과 충돌하며 오해를 유발한다. 모달 내 폴더·버전을 현재 화면에 **고정(lock)**하는 방향이 더 직관적일 수 있다. 종속 항목의 기본값을 설정하는 방식을 우선 적용했고, 흐름의 자연스러움을 위해 완전히 고정하는 방안도 검토했다.
- **선택지 최소화**: 지나치게 많은 선택지는 오히려 단점이다. 사용자가 고민 없이도 다양한 커스텀이 가능하도록 선택지 범위를 적절히 제한한다.
  > "선택지와 상황을 많이 줌으로써 사용자가 고민을 해야 된다면 오히려 그건 단점일 수 있다."
- **하이라이트 모호성**: 현재 선택된 버전의 하이라이트가 색상·시인성 면에서 불명확하다. 선택 버전 정보를 **헤더 형태로 명시**하고, 하단의 버전 정보·업로드 날짜 같은 부가 정보를 활용해 직관성을 높인다.
- **업로드 접근성**: 플러스(+) 버튼을 통해서만 업로드가 가능한 구조는 부자연스럽다. 플러스 버튼을 제거하고 **우측 상단에 업로드 버튼을 직접 배치**하는 방안이 논의되었다. 다만 분기 생성 후 플러스 버튼이 계속 추가되는 구조는 분기가 많아질수록 시각적으로 복잡해지므로, 하이라이트 기준에 따라 항목이 접히는 방식도 검토되었다. 세림 님이 제안한 **분기 기능을 완전히 분리하는 방식**도 논의되었다.
- 모달 내에서 항목 추가 시 즉시 이동시키는 방식은 경로 확인이 어렵다는 단점이 있다.
- **파일 업로드 기본값**: 동일한 이름의 파일이 업로드되면 최신 버전이 기본 선택되고, 메인 히스토리 노드가 기본으로 선택되도록 한다. 파일 업로드 위치를 강제할지, 마지막으로 열린 화면에 업로드할지 검토가 필요하다. 궁극적으로는 사용자가 메인 화면을 직접 선택하는 것조차 불필요한 작업으로 보고, **업로드만으로 최적 화면이 자동 구성되도록** 자동화하는 방향을 지향한다.
- **문서 뷰어**: 현재는 미리보기가 없어 원본 파일을 다운로드하는 방식이지만, 원본을 직접 확인할 수 있는 문서 뷰어를 도입할 계획이다. 문서 상세 화면 상단의 버전 정보 외에 하단에 배치할 정보 항목도 검토한다.
- 멘토는 UX를 국소적으로 수정하기보다 **전체 UX의 개념적 흐름(맥)을 먼저 정립**해야 일관성 있는 설계가 가능하다고 조언했다. 구현 방식을 먼저 고민하면 정책과 UX가 뒤섞여 혼란이 가중된다.

## 메인 스트림과 버전 비교 방식

- 분기가 여러 개일 때 어느 것이 **메인 스트림**인지 사용자가 인지하기 어렵다. 현재 UI는 자연수만 표시되어 판단이 어려우므로, 메인 스트림을 결정하는 명확한 기준과 이를 직관적으로 보여줄 UI가 필요하다. 사용자에게 메인 스트림 개념을 인지시켜 해당 위치로 업로드를 유도하는 정책도 요구된다.
- **변경 요약 기준**: 변경 요약은 동일 스트림 내 직전 버전을 기준으로 작성하는 것을 원칙으로 하며, v1은 요약이 없는 것으로 간주해 화면에서 제외한다. 메시지 전달 기능은 분기 간에는 적용되지 않고 직전 연결된 스트림 내에서만 작동한다. 스트림을 하나로 유지하는 편이 안정성·직관성에 유리하다는 의견이 있었다.
- **비교 대상 선택**: 이전 노드끼리 비교할지 최신 노드끼리 비교할지 선호가 갈렸다. 문서 병합 시에는 모든 분기의 **최신 노드끼리 비교**하는 기능이 더 자주 쓰일 가능성이 있다. 사용자가 직접 v2·v4 등 원하는 버전을 선택해 비교·요약 결과를 확인하는 기능도 제안되었다(미리보기 드롭다운, 기존 파일 비교 도구 인터페이스 참고).
- v1-2와 v3을 비교하거나 v4로 확장되는 과정에서 분기 병합·분리가 발생할 가능성이 높아, 사용자가 직접 분기를 선택하는 구조는 복잡성을 높이는 요인이 된다.
- 최신 버전과 지속적으로 비교하면 UI 교체 시마다 전체 과정을 반복해야 해 비효율적이다. **최신 버전 비교는 간헐적으로 수행하되 이전 버전과의 비교를 우선**하는 방향이 제시되었다.
- **최신본 기준 병합 vs 개별 작성**: 공동으로 수정·합치는 경우에는 최신본 기준 병합이 빈번하지만, 정해진 양식에 따라 개별 작성하는 경우에는 그 필요성이 낮다. 두 방식 모두 필요하다.

## 타겟 사용자 재정의

분기 기능의 필요성과 범위를 정하려면 **타겟 사용자를 명확히 정의**해야 한다. 타겟이 모호해 기능 정의와 구현 방향이 계속 흔들린다는 진단이 내려졌다.

> "타깃이 명확해야 일반화를 시킬 수 있다. 지금 일반화를 못 시켜서 이게 복잡한 것이다."

논의된 두 후보:

- **단독 작업자** — 품의서, 기획서 등 업무 문서를 혼자 작성하고 버전을 관리하는 사람. (품의서는 자유 양식의 백지 형태 기획서로, 업무 주체·일정·예산 요청·프로젝트 착수 등을 자유롭게 기재한다.)
- **공동 작업자** — 정부 지원 사업 계획서, 팁스(TIPS) 지원서 등 여러 명이 협업해 하나의 문서를 완성하는 사람.

관련 논의:

- 출장 보고서·지출 증빙·연차 보고 등 정형 업무는 ERP·사내 시스템의 폼 입력으로 대체되고 있다. 스타트업에서는 문서 작성 업무가 거의 사라졌고, 개발 외 부서에서는 구글 드라이브로 문서를 작성하는 사례가 많다. 잠재 사용자는 개발자가 아닌 일반 부서 구성원으로 상정한다.
- 브랜치 필요성은 실제 **동시 문서 편집 빈도**로 판단해야 한다. 대규모 단순 업무 조직에서는 동시 편집이 거의 없고, 스타트업 같은 소규모 조직에서는 빈번하다. IR 자료·정부 지원 사업 계획서 등 주요 문서는 공동 작성이 많고, 카카오뱅크 같은 기업에서도 협업이 잦다.
- 초기 기능은 **핵심 유저의 작업 방식**을 분석해 우선 구현한다. 문서 작업량이 많고 협업이 잦은 연구 분야(정부 사업 계획서 작성자)를 초기 타겟 후보로 두었다. 논문 같은 특수 사례는 제외하고 실무에서 빈번한 일반 케이스를 중심으로 구체화한다.
- **개인용 품의서 작성**과 **정부 계획서 공동 작성** 중 하나를 선택해 집중해야 한다. 페르소나를 지나치게 좁히기보다 공동/단독 저작의 본질적 차이를 고려해 단계적으로 접근하고, 한 타겟을 먼저 확실히 만족시킨 뒤 확장하는 전략을 권고했다. 동시 저작 기능이 우선순위가 높다고 판단되면 단계적 접근 대신 이를 최우선으로 개발한다.

## 공동 작업 워크플로우 (팁스·정부과제 사례)

멘토는 팁스(TIPS) 지원 경험을 예시로 실제 공동 작업 워크플로우를 설명했다.

- 세 명의 공동 창업자가 함께 작성했으며, 팁스 지원 문서는 **PPT 형식**이 요구된다. 사업 계획서 내 기술 문서 비중이 약 **70%**, 나머지 **30%**는 사업 계획·시장 설명·자금 운용·팀 운영·성장 전략으로 구성된다.
- 팀원이 각자 전문 영역의 핵심 내용을 먼저 작성한 뒤 대표가 취합·보완하는 방식으로 완성한다. 투자 유치 직후 **이틀**이라는 촉박한 기한에 급하게 작성했고, 수정 시마다 버전 번호를 올리고 작성자 이니셜을 표기하는 방식으로 버전을 관리했다.
- PPT로 제출해야 하는 환경에서는 변환·재포맷팅 번거로움 때문에 구글 독스 대신 PPT를 쓴다. 구글 독스 공동 작업은 파일을 전용 형식으로 변환해야 하고, 워드로 되돌리면 원본 포맷이 훼손된다. 40~100장에 달하는 대규모 문서에서 양식을 바꾸는 작업은 시간·노력이 크므로, **원본 양식 유지가 필요한 문서라면 기존 편집 방식을 고수**한다.
- 따라서 공동 작업 공간은 텍스트를 취합하는 용도로 쓰고, 포맷팅이 완료된 뒤에는 한 명의 담당자가 최종적으로 합치는 방식이 적절하다. 과거에는 사무실이 없어 각자 자택에서 작성하며 **구글 행아웃을 24시간 켜두고** 실시간 대화하고, 구글 드라이브로 새 버전을 즉시 공유했다.
- 구글 드라이브에서 직접 편집하면 임시 파일 생성으로 파일이 손상될 위험이 있어, **로컬에서 편집을 완료한 뒤 최종 파일을 업로드**하는 방식을 권장한다. 사용자 거부감을 줄이기 위해 로컬 저장 시 자동으로 전개되는 기능을 도입하고 업로드 과정을 간소화한다.
- **정부 지원 사업·R&D 과제**: TIPS를 포함한 정부 지원 사업과 대학·기업 R&D 과제는 문서 포맷·비용 산정·지원 시스템이 거의 동일하고, 국제·국내 R&D 모두 결과적으로 한글 문서 기반이다. 연구 과제는 기술 문서 성격이 강해 연구 책임자나 소수 인원이 초안 작성 후 수정하는 방식이라 브랜칭은 불필요할 가능성이 높다. 다만 국제 대규모 과제는 다수 기업·학교가 공동 지원해 기관별 문서를 통합하며, 엑셀·워드·한글이 혼재해 복잡도가 높다. 산하 기관들은 하나의 엑셀 파일에 여러 시트를 두고 개별 편집분을 취합해 1년간 관리하는 등 엑셀 활용이 매우 빈번하다. 스프레드시트와 한글 문서 간 연동 구조를 이해하려면 국제 과제 공고 양식을 내려받아 테스트 데이터를 입력해 보는 것이 효과적이다.
- 현 시스템은 파일을 개별 업로드하는 구조라 실시간 공동 작업이 어렵고, 사용자들은 구글 닥스 같은 외부 도구를 선호할 가능성이 높다. 이 때문에 해당 기능은 **1인 사용자 중심 UX**로 설계하는 것이 적절하다는 결론이 나왔다(합쳐진 파일 업로드는 지원하되 개별 수정 파일 업로드 방식은 사용하지 않는다).

## 시맨틱 Diff 기능의 범위 조정

- 시맨틱 diff(의미론적 변경점 요약)는 원래 실제 문서 diff 구현이 어려운 **MVP 단계의 대안**으로 도입된 기능이다. 다만 실제 diff를 대체하려고 넣은 것은 아니며(오타니가 직접 추가한 것도 아님), 풀 디프(Full Diff)를 위한 초기 구현 단계의 일부다.
- 시맨틱 diff는 화면 변경을 단순히 보여주는 것을 넘어 **그 의미를 설명**하는 기능이다. 사용자에게 유용할 수 있으나, 문서가 길거나 변경량이 많으면 **LLM 추론 비용과 품질 저하** 문제가 발생한다. 포인트 변경 같은 사소한 국소 수정이 많을수록 의미론적 설명 문장을 구성하기 어렵다.
- **배치(사전 처리) vs 실시간 추론 트레이드오프**: 매번 업로드 시 배치로 미리 생성하면 조회는 빠르지만, 분기·버전이 많아지면 **M×N 비교 조합**으로 비용이 급증한다. 실시간 추론은 매번 느린 경험을 노출한다.
  > "매번 새 것만 눌러서 볼 거니까 매번 느린 것만 사용자는 보게 된다."
- 사용자가 한 번 확인한 시맨틱 diff를 반복 조회할 가능성은 낮아 캐싱 효과도 제한적이다. 편집을 마친 사용자는 최신 정보로 확장하려는 경향이 있어 기존 정보를 다시 보지 않는다. 시맨틱 기능은 자신이 편집 중인 문서보다 **오래된 데이터나 타인의 작업물을 검토**할 때 필요성이 더 높다.
- **LLM 청크 분할 상충**: 청크를 작게 자르면 속도는 빨라지나 의미적 연결성이 떨어지고, 한 번에 넣으면 느려지는 선택의 문제가 있다.
- 논의 결과, 시맨틱 diff를 항목별로 세세히 나열하기보다 **버전 간 변경점을 간결하게 요약(Summary)하는 형태**로 재정의하기로 합의했다. 전체를 빠르게 훑는 것이 목적이므로 핵심 의미를 압축 전달한다.
- **파일 간 diff**(1월 보고서 vs 2월 보고서처럼 서로 다른 문서 비교)는 버전 관리와 성격이 다르며, 현재 범위에서 제외하고 **추후 확장 기능으로 분리**한다. 버전 간 diff는 필수로 우선 개발하고, 문서 간 diff는 그 이후 상대적으로 수월하게 구현할 수 있을 것으로 본다.
- MVP 뷰어 화면에는 diff 정보를 표시하지 않으며, 확장 기능 사용 시 규칙에 따라 표시한다. MVP 단계에서는 확장성을 고려하지 않고 기획서에 명시된 기능 구현에 집중한다.
- 관련 UX 아이디어: 변경 섹션·버전별 정보를 **카드 형태**로 제시하고 카드 클릭 시 해당 위치로 스크롤, 두 문서를 나란히 배치해 미리보기하는 별도 화면·버튼. AI 기능은 사용자가 선택적으로 요청할 때만 제공하며, MVP에서 AI로 CSS 포함 파일 파싱이 가능하고 40페이지 이상 대용량 파일도 처리 가능함을 확인했다. (V2→V3 전환 시 네 가지 변경 포인트와 시맨틱 예시 화면의 의도, JSON 예시 미확인 건은 추가 확인이 필요하다.)

## 정책 문서화와 용어집

멘토는 팀 내에 **기능·정책에 대한 공통 기준 문서가 없어** 논의할 때마다 각자 이해가 달라진다고 지적했다. 구두 정책은 해석 차이와 기억 변화로 정확성이 떨어지므로 문서화가 필요하다.

- **정책 문서 = 헌법**: 각 기능이 어떻게 동작하는지, 어떤 값·조건이 있는지를 한 줄씩 약관처럼 명확·간결하게 명시한다. MD 파일로 관리하고 **문서에 없는 기능은 개발하지 않는 것**을 원칙으로 한다. 변경 시 사이드 이펙트를 고려해 반드시 사전 팀 논의를 거친다. 한 번에 완성되는 것이 아니라 초안 작성 후 정기 검토로 지속 업데이트한다. 사용자 문의·기능 요구에 정책 기반으로 일관되게 응대하기 위한 표준 문서이자, 설계부터 CS 대응·개발까지 전 과정의 기준점이다.
- **용어집(Glossary) = 도메인 문서**: 히스토리·버전·분기·노드·폴더·스트림 등 핵심 용어를 정의한 글로서리다. 정책 문서는 이 도메인 문서를 바탕으로 실제 시스템 동작 규칙을 기술한다. 계약서·법령의 용어 정의 섹션처럼, 순서·표기가 미세하게 달라도 의미가 달라질 수 있으므로 사전에 일관되게 정의한다.
- **화면 표시 용어 ↔ 기술 용어 매핑**: 두 명칭이 다르더라도 의미적 매칭은 반드시 필요하다. 예를 들어 '이력·히스토리·로그'는 맥락에 따라 다르게 쓰인다. 히스토리는 사용자 관점의 과거 이력, 로그는 기술 영역의 기록에 가깝다. 시스템의 '로그'는 관성상 바꾸기 어렵지만, 사용자에게는 '히스토리'가 훨씬 직관적이므로 이를 활용한다. 마케팅·설명 목적으로 일상 용어를 차용할 경우 화면 표시용임을 명확히 하고, 표기 간 매칭 관계를 이력으로 기록한다.
- **트리 vs 그래프**: 히스토리 화면의 구조물 명칭(트리·그래프·분기표 등)은 원형 요소가 없다는 점을 고려해 기술 용어에 얽매이지 않고 정한다. 기술적으로는 데이터 구조를 트리로 구현해야 하며, 그래프는 서클(순환 참조)을 가정하는 구조다. 현재 시스템에서는 서클이 발생하지 않으므로 그래프의 일부인 트리 형태로 본다.
- 서비스 규모가 커지면 용어 불일치로 혼선·갈등이 생기므로, 코드와 서비스 전반에서 용어 일관성을 유지한다.

## 개발 우선순위와 설계 원칙

- **분기 기능은 유지하되 우선순위를 낮춘다.** 사용자 경험에 도움은 되지만 필수는 아니므로, 복잡성을 줄이기 위해 먼저 **메인 스트림(리니어 버전 관리) 기능 완성**에 집중하고 이후 단계적으로 확장한다.
- 어려운 문제를 단순·명확하게 해결하는 것이 좋은 제품의 핵심이다. 기술적 난관을 넘어 정책적 해결 방안을 찾고 이를 기술로 구현하는 것이 개발자의 역할이다.
- 구현 난이도를 기준으로 기능을 쳐내는 가지치기는 지양하고, **핵심 기능에 집중하는 방식**으로 가지치기한다. 모든 예외를 수용하는 커스텀 시스템이 아니라 일반적인 사용 사례를 중심으로 설계하고, 다양한 예시 중 일반화 가능한 핵심 사례를 선별한다.
- 기능 정의에 앞서 충분한 논의로 기준을 세우고, 기능별로 **사용자에게 제공할 정보와 요구할 정보(입출력)**를 명확히 정의한다. UI 화면 구성보다 입출력 정의를 우선한다.
- **그림(화면 설계)보다 글(정책 줄글)로 먼저 기능을 정의**한다. 복잡한 기능을 한 번에 구현하기보다 리니어한 버전 관리부터 빠르게 완료한 뒤 점진적으로 확장한다.
- UI·UX 디테일보다 **핵심 코어 기능 완성에 집중**한다. UX 디테일은 사례를 바탕으로 점진 보완하는 것이 효율적이며, 이번 달까지는 핵심 기능 구현을 최우선으로 둔다. 사용자의 **고민·클릭·입력을 최소화**하는 것을 설계 원칙으로 삼는다.
- 정책이 수시로 바뀌고 의견이 개별적으로 제시돼 정리되지 않는 문제가 있으므로, 정책을 명확히 수립한 후 단계적으로 보완하는 의사결정 체계로 개선한다. 이미 혼재되어 진행된 업무가 있으므로 정책 범위를 축소하더라도 내용은 명확히 정의한다.

## 결정 사항 (Action Items)

- **정책 문서 초안 작성** — 분기·버전·히스토리·Diff 등 핵심 기능의 동작 방식을 한 줄씩 명문화해 팀 공통 기준을 수립한다.
- **용어집(Terminology) 정리** — 서비스 내 사용 용어(한글·영문)를 정의하고, 화면 표기명과 기술 용어 간 매핑을 확정한다.
- **타깃 유저 재정의** — 개인 문서 작성자와 공동 저작자 중 1차 타깃을 명확히 선택하고, 해당 유저의 주요 사용 시나리오(User Scene)를 문서화한다.
- **분기 기능 정책 확정** — 분기 생성 기준(최신 vs 과거 버전), 허용 분기 수, Merge 미지원 시 사용자 안내 방식을 정책 문서에 명시한다.
- **Semantic Diff → 변경점 요약으로 재정의** — 전체 변경 요약(Summary) 기능으로 범위를 축소하고, 실제 문서 Diff는 추후 확장 기능으로 분리 계획을 세운다.
- **모달 UX 고정 방식 검토** — 업로드 모달 내 폴더·버전 선택을 현재 화면에 고정하는 방안을 구체화하고, 변경이 필요한 경우의 대안 흐름을 설계한다.
- **핵심 코어 기능 완성 집중** — 리니어 버전링 → 분기 → Diff 순서로 스코프를 명확히 하고, 현 단계에서는 UI 디테일보다 기능 완성도에 집중한다.
