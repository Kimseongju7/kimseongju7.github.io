OpenAI 팀이 5개월간 **사람이 코드를 한 줄도 직접 쓰지 않고** Codex만으로 약 100만 라인짜리 내부 베타 제품을 구축·출시한 실험기다. 엔지니어의 일은 코드 작성에서 벗어나, 에이전트가 안정적으로 일할 수 있는 환경·의도·피드백 루프를 설계하는 "하네스 엔지니어링"으로 바뀐다.

## 실험 개요: 사람이 조정하고, 에이전트가 수행

- 내부 일일 사용자와 외부 알파 테스터가 있는 실제 제품으로, 출시·배포·장애·복구가 모두 진행됐다.
- 애플리케이션 로직, 테스트, CI 구성, 문서화, 관측 가능성, 내부 툴링까지 모든 코드가 Codex로 작성됐다. 수작업 대비 약 1/10의 시간이 걸린 것으로 추정.
- 2025년 8월 말 빈 git 리포지터리에서 시작했다. 초기 스캐폴드(리포지터리 구조, CI, 서식 규칙, 패키지 관리자, 앱 프레임워크)와 최초의 AGENTS.md까지 GPT-5 기반 Codex CLI가 생성했다.
- 5개월 후 리포지터리는 약 100만 라인. 엔지니어 3명이 약 1,500개의 PR을 열고 병합했으며(1인당 하루 평균 3.5개), 팀이 7명으로 커지자 처리량이 오히려 증가했다.
- 핵심 철학: **수동으로 작성한 코드가 없다(no hand-written code).**

## 엔지니어의 역할 재정의

- 초기 진행이 더뎠던 이유는 Codex의 역량 부족이 아니라 환경 미비였다. 에이전트에게 도구·추상화·내부 구조가 부족했던 것.
- 엔지니어의 주 업무는 "어떤 기능이 누락됐고, 에이전트가 읽기 쉽고 실행 가능하게 만들려면 어떻게 해야 하나?"를 고민하는 것이 됐다.
- 사람은 거의 전적으로 프롬프트로 시스템과 상호작용한다. Codex에게 로컬에서 자체 변경을 검토시키고, 로컬·클라우드의 에이전트 리뷰를 추가로 요청하고, 모든 에이전트 검토자가 만족할 때까지 반복시킨다(사실상 랄프 위검 루프, Ralph Wiggum Loop).
- 사람의 PR 리뷰는 가능하지만 필수가 아니며, 거의 모든 검토가 에이전트 간 처리로 전환됐다.

## 애플리케이션을 에이전트가 읽을 수 있게 만들기

- 인적 QA가 병목이 되자, 앱 UI·로그·메트릭을 Codex가 직접 읽고 이해할 수 있게 만들었다.
- git worktree별로 앱을 부팅할 수 있게 해 변경마다 인스턴스 하나를 Codex가 실행·제어한다.
- Chrome DevTools Protocol을 에이전트 런타임에 연결하고 DOM 스냅샷·스크린샷·탐색용 스킬을 만들어, 버그 재현·수정 검증·UI 동작 추론이 가능해졌다.
- 로그·메트릭·추적은 워크트리별 일시적 로컬 관측 스택으로 노출된다. 에이전트는 LogQL로 로그를, PromQL로 메트릭을 쿼리한다. 덕분에 "서비스 시작 800ms 이내", "핵심 사용자 여정 4개에서 어떤 스팬도 2초 초과 금지" 같은 프롬프트가 다루기 쉬워졌다.
- 한 번의 Codex 실행이 6시간 이상(종종 사람이 자는 동안) 지속되는 경우가 많다.

## 리포지터리 지식을 기록 시스템으로

- 핵심 교훈: **Codex에 1,000페이지 설명서가 아니라 맵을 제공하라.**
- "하나의 거대한 AGENTS.md" 접근은 예측 가능하게 실패했다:
  - 컨텍스트는 희소 자원이라 거대 지침 파일이 핵심 제약을 놓치게 만든다.
  - 지침이 너무 많으면 지침이 아니다 — 모든 것이 "중요"하면 아무것도 중요하지 않다.
  - 거대 매뉴얼은 금세 낡은 규칙의 무덤이 된다.
  - 단일 블롭은 기계적 점검(커버리지·신선도·소유권·교차 연결)이 어렵다.
- 그래서 AGENTS.md를 백과사전이 아닌 **목차**로 취급한다. 약 100라인의 짧은 AGENTS.md가 맵 역할을 하고, 실제 지식은 구조화된 `docs/` 디렉터리(design-docs, exec-plans, product-specs, references, 생성 문서 등)에 기록 시스템으로 존재한다.
- 계획은 일급 아티팩트다. 복잡한 작업은 진행 상황·의사결정 로그와 함께 실행 계획으로 리포지터리에 버전 관리된다.
- 전용 린터와 CI가 지식 베이스의 최신성·교차 링크·구성을 검증하고, 반복 실행되는 "doc-gardening" 에이전트가 낡은 문서를 찾아 수정 PR을 연다.

## 에이전트 가독성이 목표

- 에이전트 관점에서 실행 중 컨텍스트로 접근할 수 없는 것은 사실상 존재하지 않는다. Google Docs, 채팅 스레드, 사람 머릿속 지식은 접근 불가 — 리포지터리 내 버전 관리 아티팩트만 유효하다.
- Slack에서 합의된 아키텍처 패턴도 에이전트가 검색할 수 없으면 3개월 뒤 입사한 신입에게 없는 지식과 같다.
- 리포지터리 안에서 완전히 내부화·추론 가능한 종속성과 추상화를 선호한다. "지루한" 기술이 에이전트가 모델링하기 쉽다.
- 때로는 공개 라이브러리의 불투명한 동작에 맞추는 것보다 기능 일부를 재구현하는 편이 쌌다. 예: 범용 `p-limit` 스타일 패키지 대신 OpenTelemetry 계측과 통합되고 테스트 커버리지 100%인 자체 map-with-concurrency 헬퍼를 구현.

## 아키텍처와 취향을 기계적으로 강제

- 문서만으로는 일관성이 유지되지 않는다. **구현을 세세하게 관리하는 대신 불변 조건(invariant)을 강제한다.** 예: 경계에서 데이터 형태를 파싱하라고 요구하되 특정 라이브러리는 지정하지 않음(모델은 Zod를 선호하는 듯).
- 각 비즈니스 도메인은 고정된 레이어(Types → Config → Repo → Service → Runtime → UI)로만 전달되고, 교차 관심사(인증·커넥터·텔레메트리·기능 플래그)는 Providers라는 명시적 인터페이스로만 유입된다. 그 외는 전부 금지이며 Codex가 만든 맞춤 린터와 구조적 테스트로 기계적으로 강제된다.
- 맞춤 린트로 구조화 로깅, 명명 규칙, 파일 크기 제한, 플랫폼 안정성 요구를 정적으로 강제하고, 린트 오류 메시지에 수정 지침을 담아 에이전트 컨텍스트에 주입한다.
- 이런 아키텍처는 보통 엔지니어 수백 명 규모까지 미루는 것이지만, 코딩 에이전트에게는 초기 전제 조건이다. 제약이 있어야 드리프트 없이 속도를 낼 수 있다.
- 결과 코드가 인간의 문체 취향과 다를 수 있지만, 정확하고 유지관리 가능하며 이후 에이전트 실행에서 읽기 쉬우면 기준을 충족한다. 인간의 취향은 리뷰 코멘트·리팩터링 PR·버그를 통해 문서와 툴링으로 다시 인코딩된다.

## 처리량이 바꾸는 병합 철학

- 리포지터리는 최소한의 차단 병합 게이트로 운영된다. PR 수명이 짧고, 테스트 불안정성(flakiness)은 진행을 막기보다 후속 실행으로 해결한다.
- 에이전트 처리량이 사람의 주의력을 훨씬 초과하는 시스템에서는 수정이 싸고 대기가 비싸다. 처리량이 적은 환경에는 맞지 않는 절충이다.

## "에이전트 생성"의 실제 범위와 자율성

- 에이전트가 생성하는 것: 제품 코드·테스트, CI 구성·릴리스 툴링, 내부 개발자 도구, 문서·설계 내역, 평가 하네스, 리뷰 코멘트·응답, 리포지터리 관리 스크립트, 프로덕션 대시보드 정의까지 전부.
- 개발 루프(테스트·검증·리뷰·피드백·복구)가 시스템에 인코딩되면서, Codex가 한 번의 프롬프트로 기능을 처음부터 끝까지 주도하는 임계점을 넘었다: 현재 상태 검증 → 버그 재현 → 실패 상황 동영상 녹화 → 수정 구현 → 앱 실행으로 검증 → 해결 동영상 녹화 → PR 열기 → 피드백 응답 → 빌드 실패 감지·수정 → 판단 필요 시에만 사람에게 에스컬레이션 → 병합.
- 단, 이 동작은 이 리포지터리 특유의 구조·툴링에 크게 의존하며, 유사한 투자 없이 일반화된다고 가정하면 안 된다.

## 엔트로피와 가비지 컬렉션

- 완전한 자율성은 새 문제를 낳는다. Codex는 리포지터리의 기존 패턴(최적이 아닌 것 포함)을 복제하므로 시간이 지나면 드리프트가 필연적이다.
- 초기에는 팀이 매주 금요일(일주일의 20%)을 "AI 슬로프" 청소에 썼지만 확장되지 않았다.
- 대신 "황금 원칙"을 리포지터리에 직접 인코딩했다. 예: (1) 직접 만든 헬퍼보다 공유 유틸리티 패키지 선호, (2) "YOLO 스타일" 데이터 탐색 금지 — 경계 검증 또는 유형 지정 SDK에 의존.
- 정기적으로 편차를 검사하고 품질 등급을 갱신하며 대상 리팩터링 PR을 만드는 Codex 백그라운드 작업을 운영한다. 대부분 1분 이내 검토 후 자동 병합된다.
- 이는 가비지 컬렉션처럼 작동한다. 기술 부채는 고금리 대출과 같아, 조금씩 꾸준히 갚는 편이 낫다.

## 여전히 배우고 있는 것

- 완전한 에이전트 생성 시스템에서 아키텍처 일관성이 수년에 걸쳐 어떻게 진화하는지는 아직 모른다.
- 분명해진 것: 소프트웨어 구축에는 여전히 규율이 필요하지만, 그 규율은 코드보다 스캐폴딩에서 드러난다.
- 현재 가장 어려운 과제는 에이전트가 복잡하고 안정적인 소프트웨어를 대규모로 구축·유지하도록 돕는 환경·피드백 루프·제어 시스템의 설계다.
