## 한 줄 정의
HTML 문서 하나만 처음에 불러오고, 이후 페이지 이동은 새로고침 없이 JavaScript가 화면 내용만 바꿔치기하는 웹 애플리케이션 방식.

## 자세히
SPA(Single Page Application, 단일 페이지 애플리케이션)는 이름 그대로 **웹사이트 전체가 하나의 HTML 페이지**로 돌아가는 구조다. 전통적인 웹([MPA](/posts/mpa/), Multi Page Application)에서는 링크를 누를 때마다 브라우저가 서버에 새 HTML을 요청하고, 응답이 오면 화면 전체를 다시 그린다. 그래서 페이지를 옮길 때마다 하얗게 깜빡이는 새로고침이 생긴다. SPA는 이걸 없앤다. 최초에 빈 껍데기 HTML과 [JavaScript 번들](/posts/js-bundle/)을 한 번 받아오고, 그다음부터는 사용자가 어디를 누르든 서버에 **HTML을 다시 요청하지 않고**, JS가 필요한 데이터(주로 JSON)만 API로 받아와 화면의 바뀔 부분만 갈아끼운다.

이 방식은 [CSR(클라이언트 사이드 렌더링)](/posts/frontend-study/)과 짝을 이룬다. 브라우저가 거의 비어 있는 HTML을 받고, JS가 실행되면서 화면을 구성하기 때문이다. 장점은 명확하다. 최초 로딩 이후 **페이지 전환이 매우 빠르고**(네트워크 왕복 없이 즉시 화면이 바뀜), 서버는 데이터만 내려주면 되니 부하가 적으며, 데스크톱 앱처럼 매끄러운 사용자 경험을 만들 수 있다. 그래서 Gmail, Notion, 대시보드, 웹 에디터처럼 상호작용이 많고 화면 안에서 계속 머무는 서비스에 잘 맞는다.

단점도 뚜렷하다. 첫째, **첫 화면이 뜨기까지 느리다**. JS 번들을 다 받아서 실행해야 비로소 화면이 그려지므로, 번들이 크면 초기 로딩이 답답하다. 둘째, **[SEO](/posts/seo/)에 불리하다**. 검색엔진 크롤러가 페이지를 방문했을 때 JS가 실행되기 전이면 텅 빈 HTML만 보게 되어, 내용이 색인되지 않을 수 있다. 셋째, 브라우저의 뒤로가기·주소창·북마크가 원래는 페이지 단위로 동작하는데, SPA는 화면이 하나뿐이라 이걸 **라우팅 라이브러리**(React Router 등)로 흉내 내야 한다.

이 SEO·초기 로딩 문제를 풀기 위해 나온 게 [SSR](/posts/ssr/)(서버 사이드 렌더링)이나 SSG(정적 사이트 생성)를 SPA와 섞는 방식이다. 서버가 완성된 HTML을 먼저 내려줘서 첫 화면과 SEO를 챙기고(=MPA의 장점), 그 뒤에 JS가 붙어(hydration) 이후 전환은 SPA처럼 빠르게 동작하게 한다. Next.js·Nuxt 같은 프레임워크가 페이지별로 이 둘을 골라 쓸 수 있게 해 주는 이유다. 참고로 이 블로그가 쓰는 [Jekyll·Chirpy](/posts/jekyll-chirpy/)는 반대편 극단인 순수 정적 사이트(SSG)로, 검색·초기 로딩에 강한 대신 SPA 같은 앱스러운 상호작용은 목표로 하지 않는다.

## 예시
- **대표 서비스**: Gmail, Google Maps, Notion, Trello, Figma — 한 화면 안에서 계속 상호작용하는 앱들이 SPA다.
- **동작 흐름**: 사용자가 SPA에서 "설정" 탭을 누르면 → 브라우저는 서버에 새 페이지를 요청하지 않고 → JS가 `/api/settings`에서 JSON만 받아와 → 화면의 본문 영역만 설정 화면으로 교체한다. 주소창의 URL은 라우터가 `/settings`로 바꿔 주지만 실제 새로고침은 없다.
- **주요 프레임워크**: React, Vue, Angular, Svelte — 대부분 [컴포넌트](/posts/component/) 단위로 화면을 조립하고 상태에 따라 다시 그리는 SPA 개발을 전제로 한다. [React Doctor](/posts/react-doctor/)가 검사하는 대상도 이런 React 기반 SPA 코드다.
