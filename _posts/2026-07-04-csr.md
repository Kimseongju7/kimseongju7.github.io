## 한 줄 정의
CSR(Client-Side Rendering, 클라이언트 사이드 렌더링)은 서버가 거의 빈 HTML만 보내고, **브라우저에서 JS가 실행되며 화면을 그려내는** 렌더링 방식이다.

## 자세히
전통적인 웹은 서버가 완성된 HTML을 만들어 보냈지만, CSR에서는 브라우저가 처음에 뼈대만 있는 빈 `index.html`과 [자바스크립트 번들](/posts/js-bundle/)을 받는다. 이후 JS가 실행되면서 데이터를 불러오고 DOM을 조립해 화면을 완성한다. React, Vue 같은 프레임워크로 만든 [SPA](/posts/spa/)(Single Page Application)가 대부분 이 방식이다.

**장점**은 첫 로딩 이후의 사용자 경험이다. 한 번 JS가 올라오면 페이지를 이동할 때 서버에 전체 HTML을 다시 요청하지 않고, 필요한 데이터만 받아 화면의 일부만 바꿔치기한다. 그래서 새로고침 없이 전환이 매우 빠르고, 서버는 HTML을 매번 그려줄 필요가 없어 부하가 적다.

**단점**은 두 가지다. 첫째, **첫 화면이 뜨기까지 느리다** — 빈 HTML을 받은 뒤 JS를 내려받고 실행하고 데이터를 불러와야 비로소 무언가 보이기 때문이다. 둘째, **[SEO](/posts/seo/)에 불리하다** — 검색 엔진 크롤러가 페이지를 가져갔을 때 JS 실행 전이라면 빈 화면만 보게 되어 콘텐츠가 색인되지 않을 수 있다. 그래서 블로그·랜딩 페이지처럼 검색 노출이 중요한 곳은 [SSR](/posts/ssr/)(Server-Side Rendering)이나 SSG(정적 사이트 생성)를 쓰고, SEO가 중요치 않은 관리자 대시보드 같은 내부 화면은 CSR로 충분하다.

Next.js 같은 [메타프레임워크](/posts/meta-framework/)는 페이지별로 CSR·SSR을 섞어 쓸 수 있어, 화면 성격에 따라 최적의 방식을 고를 수 있다. 참고로 이 블로그([Jekyll·Chirpy](/posts/jekyll-chirpy/))는 빌드 시점에 HTML을 미리 만들어 두는 **SSG** 방식이라, CSR과 반대로 첫 로딩과 SEO에 강하다.

## 예시
```
CSR 흐름:
브라우저 → 서버   : GET /
서버   → 브라우저 : 빈 <div id="root"></div> + app.js
브라우저          : app.js 실행 → API 호출 → 화면 렌더링

SSR 흐름:
브라우저 → 서버   : GET /
서버              : HTML 완성해서
서버   → 브라우저 : 이미 내용이 채워진 HTML (이후 JS가 붙어 상호작용)
```
