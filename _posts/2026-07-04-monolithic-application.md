## 한 줄 정의
애플리케이션의 모든 기능(UI·비즈니스 로직·데이터 접근)을 하나의 코드베이스·하나의 배포 단위로 묶어 통째로 빌드·배포·실행하는 아키텍처 방식.

## 자세히
"모놀리스(monolith)"는 원래 "하나의 큰 돌덩어리"라는 뜻이다. 소프트웨어에서는 회원가입, 결제, 알림, 관리자 화면 같은 서로 다른 기능들이 각각 나뉘어 있더라도 결국 **하나의 프로세스로 컴파일·배포되고, 하나의 서버에서 함께 실행되는** 애플리케이션을 가리킨다. 내부적으로는 계층(controller / service / repository)이나 모듈로 코드를 나눌 수 있지만, 배포 관점에서는 `.jar` 하나, 컨테이너 이미지 하나처럼 **쪼갤 수 없는 단일 덩어리**로 다뤄진다.

가장 큰 장점은 **단순함**이다. 코드가 한 곳에 있어 개발·디버깅·테스트가 직관적이고, 함수 호출로 기능이 바로 이어지므로 네트워크 통신 오버헤드가 없다. 배포도 "빌드해서 한 번 올리면 끝"이라 초기 개발 속도가 빠르다. 그래서 대부분의 서비스는 모놀리스로 시작하는 것이 합리적이며, 스타트업 초기나 규모가 크지 않은 서비스에는 오히려 이 방식이 더 어울린다.

한계는 **덩치가 커질수록** 드러난다. 코드베이스가 방대해지면 빌드·배포 시간이 길어지고, 작은 기능 하나를 고쳐도 전체를 다시 배포해야 한다. 특정 기능만 트래픽이 몰려도 애플리케이션 전체를 통째로 복제해서 확장(scale-out)해야 하므로 자원 낭비가 생긴다. 또 한 모듈의 장애나 메모리 누수가 프로세스 전체를 끌어내려 **장애 격리가 어렵다**. 팀이 커지면 같은 코드베이스를 여럿이 동시에 건드리며 충돌하기 쉽고, 기술 스택도 하나로 고정된다.

이런 문제를 나눠서 풀자는 것이 반대편의 **[마이크로서비스(microservices)](/posts/microservices-architecture/)** 아키텍처다. 기능별로 독립적으로 배포·확장·장애 격리가 가능한 작은 서비스들로 쪼개고, 서비스끼리는 네트워크(HTTP·메시지 큐)로 통신한다. 대신 서비스 간 통신·분산 트랜잭션·운영 복잡도라는 새로운 비용을 떠안는다. 그래서 실무에서는 "처음부터 마이크로서비스"가 아니라, **잘 모듈화된 모놀리스(modular monolith)로 시작해 병목이 명확해지는 부분만 서비스로 분리**해 나가는 접근이 널리 권장된다.

## 예시
- **모놀리스 배포**: 쇼핑몰의 상품·주문·결제·알림 로직이 모두 하나의 Spring Boot 프로젝트 안에 있고, `app.jar` 하나로 빌드되어 서버에 올라간다. 결제 코드만 고쳐도 전체를 다시 배포한다.
- **마이크로서비스로 분리**: 같은 쇼핑몰을 `상품 서비스`, `주문 서비스`, `결제 서비스`, `알림 서비스`로 나누면, 결제 로직 수정 시 결제 서비스만 다시 배포하고, 세일 때 주문 서비스만 인스턴스를 늘려 확장할 수 있다.

```text
[모놀리스]                         [마이크로서비스]
┌─────────────────────┐          ┌───────┐ ┌───────┐
│  상품  주문  결제  알림 │   →      │ 상품  │ │ 주문  │
│   (하나의 프로세스)     │          └───────┘ └───────┘
└─────────────────────┘          ┌───────┐ ┌───────┐
   배포 단위 = 1개                  │ 결제  │ │ 알림  │
                                  └───────┘ └───────┘
                                    각자 독립 배포·확장
```
