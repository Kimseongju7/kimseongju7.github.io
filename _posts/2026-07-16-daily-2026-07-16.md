## 배운 것

- 기능을 설계할 때는 판단 기준을 명확히 가지고 해야 한다. 그 기준은 타깃 사용자에서 나온다 — 타깃 사용자가 진짜 필요로 하는 기능인지, 사용자의 문제를 해결할 수 있는지로 판단해야 한다.
- 기능 설계가 모호하다면 사용자의 선택지와 고민을 줄이는 방향으로 가라. 나 역시 다른 소프트웨어의 사용자 입장에서 설정할 게 많으면 귀찮아서 안 쓰게 된다 — 이 점을 유념하고 기능과 소프트웨어를 설계해야 한다.
- 정책 문서는 꼭 작성해야 한다. 같은 기능이라도 사람마다 받아들이는 방식과 좋다고 생각하는 지점이 달라서, 구현 단계에서 혼동이 생기거나 완성 후에 다시 고쳐야 할 수 있다. 설계 단계에서 미리 상의해 확정해두는 편이 품이 덜 든다.
- 팀원과 상의할 때 회의가 산으로 가면 판단 기준을 계속 상기시킬 것. 우리의 판단 기준은 일단 이 두 가지다 — 타깃 사용자의 문제를 해결할 수 있는가, 사용자의 선택지를 최소화하는가.

**정황 진단과 실제 검증의 차이**
- ✨ `.env`에 4월이 박혀 있고 `New=0`이 계속된다는 정황만으로 버그를 확신했지만, 실제로 요청해 비교해보니 `setDate`는 무시되는 파라미터였다. 로그는 "새 공지 없음"과 "엉뚱한 달을 보고 있음"을 구분하지 못하고 둘 다 `New=0`으로 찍힌다 — 3개월간 멀쩡해 보인 이유이자, 동시에 정황만으로 판정하면 안 되는 이유다. 현상을 직접 보고 있는 쪽("잘 돌아가고 있긴 한데")이 맞을 확률이 높다.
- ✨ 테스트 전에 `--dry-run`이 상태 파일을 저장하는지부터 확인한 게 결정적이었다. `state_store.save`가 `notify` 조건 밖에 있어서 dry-run도 실제 상태를 오염시킨다 — 그냥 테스트했으면 진짜 알림이 막힐 뻔했다.

**`os.getenv(키, 기본값)`의 빈 문자열 함정**
- ✨ `os.getenv`는 키가 **없을 때만** 기본값을 쓴다. `.env`에 `KEY=`처럼 빈 값으로 두면 빈 문자열이 그대로 들어가서 기본값이 안 먹는다. 줄을 비우는 게 아니라 주석 처리해야 맞고, 아니면 `os.getenv(...) or None`으로 방어해야 한다.
- ✨ 관련해서 `Settings.from_env()`를 `while` 루프 바깥에서 한 번만 호출하면 시작 시점의 값이 프로세스 수명 내내 얼어붙는다. 몇 달씩 돌리는 데몬에서 "현재 월" 같은 걸 기본값으로 계산할 때 특히 위험한 패턴이다.

**Claude Code 세션 기록과 워크트리**
- ✨ 데스크탑 앱(Code 탭) 세션도 CLI와 동일하게 `~/.claude/projects/<슬러그>/<세션ID>.jsonl`에 로컬 저장돼서, 일지 정리(`recall-claude`)가 수정 없이 그대로 동작한다. 단 claude.ai/code(웹)는 클라우드 실행이라 로컬 기록이 전혀 안 남는다 — 일지에 남길 작업은 웹 말고 데스크탑·터미널에서 해야 한다.
- ✨ 저장 위치는 같아도 데스크탑·CLI·웹은 세션 목록을 각자 관리해서 픽커에 서로가 안 뜬다. CLI 세션을 데스크탑에서 직접 이어받을 수 없고, `/handoff-prompt`로 인계 프롬프트를 만들어 붙여넣는 방식으로 넘겨야 한다.
- ✨ git worktree는 같은 저장소를 별도 디렉터리에 하나 더 체크아웃한 것이다. 데스크탑 앱은 새 세션(⌘N)마다 `<루트>/.claude/worktrees/`에 전용 워크트리를 자동 생성하고, CLI는 `claude --worktree <이름>`, 서브에이전트는 `isolation: worktree`로 쓴다. 변경이 없으면 종료 시 자동 삭제된다.

**부하는 파일 "용량"이 아니라 "개수"가 만든다**
- ✨ Google Drive가 CPU 78%를 쓰는데 업로드는 초당 8.8KB뿐이었다. 대용량 전송은 이미 끝났고 남은 건 37만 개 파일의 메타데이터 훑기였던 것 — 파일당 고정 비용(색인·stat·동기화 상태 확인)이 개수만큼 곱해지니 전송할 바이트가 없어도 CPU는 계속 탄다. arxiv 논문처럼 작은 파일이 수십만 개면 총 용량보다 개수가 문제다.
- ✨ "용량이 많아 오래 걸리는 것"과 "헛도는 것"은 겉보기 CPU가 똑같아서, 순간 전송 속도를 직접 재봐야 갈린다(2시간 46분 평균은 초당 420KB였는데 지금은 그 1/50이었다). 그리고 arxiv가 429로 막던 구간에는 그 색인 비용만 치르고 얻는 건 0이었다 — 백오프가 정상 작동해도 서버가 건 리밋은 시간이 지나야 풀리니, 1시간 멈췄다 재시작하자 전부 `200 OK`로 돌아왔다.

**"코어 가치가 1인에서 완결되는가"를 엄격하게 적용하기**
- ✨ 싱글 유저를 먼저 검증하는 진짜 이유는 신호의 순도다. 멀티로 내면 지표가 안 나올 때 가치 가설이 틀린 건지, 상대방이 안 들어와서인지, 초대 UX가 나빠서인지 분리가 안 된다. 반증 비용도 다르다 — 싱글은 기존 기능으로 당장 검증되지만, 멀티는 권한·초대·알림·충돌 정책을 다 만들어야 시작이라도 한다.
- ✨ 도입 임계도 결정적이다. 싱글 가치는 유저 1명이 가입하는 순간 100% 발생하지만, 공동 가치는 상대방(상사·팀원)도 들어와야 완성된다 — 신뢰 자산 없는 신생 제품이 "당신 상사도 데려오세요"를 요구하면 콜드 스타트 문제를 검증 단계부터 떠안는 셈이다. 게다가 범위는 넓히긴 쉽고 좁히긴 어려워서 되돌림이 비대칭이다.
- ✨ 기준은 느슨한 버전("1인으로도 어떻게든 쓸 수 있는가")이 아니라 엄격한 버전("코어 가치가 1인에서 완결되는가")으로 기억할 것. 공유형 여행앱도 혼자 쓸 수는 있지만 무의미하다. 조직은 가치를 구성하는 게 아니라 더하는 요소라는 구분이 여기서 나온다.

**[YAGNI](/posts/yagni/) (You Aren't Gonna Need It)**
- ✨ "그거 어차피 필요 없을 거야" — 당장 필요하지 않은 기능·추상화·문서를 미리 만들지 말라는 원칙. 미리 만든 것의 상당수는 끝내 안 쓰이거나, 실제 필요가 왔을 때 요구사항이 달라 다시 만들게 되는데 그동안은 유지보수 대상으로 남아 비용만 든다. KISS·DRY와 함께 묶이는 원칙이고, "야그니"라고 읽는다("양기"가 아님 — 철자 순서가 G-N).

**Claude Code 플러그인 생태계: [우로보로스](/posts/ouroboros-agent-os/) vs superpowers vs everything-claude-code**
- ✨ 셋은 경쟁 제품이 아니라 "AI 코딩의 어디가 문제냐"에 대한 답이 다르다. 우로보로스(Q00, 한국산)는 *뭘 만들지 모르는 것*이 문제라 보고 인터뷰→시드→실행→평가의 재귀 루프로 모호성 점수·온톨로지 유사도가 임계치에 도달할 때까지 반복한다(스킬 20개, MCP 서버, Rust TUI 포함). superpowers(Jesse Vincent, Anthropic 공식 마켓플레이스 등재)는 *어떻게 만드는지가 엉성한 것*이 문제라 보고, everything-claude-code는 *도구 자체가 부족한 것*이 문제라 본다 — 로컬엔 이 중 everything-claude-code만 설치돼 있었다.

**불변성의 압력은 "push"가 아니라 "공유"에서 온다 — git 모델을 제품 정책에 적용하기**
- ✨ git은 역사를 지우지 않고 앞으로 고치는 게(revert) 정상 경로다. 중간 커밋 삭제는 존재하지 않는 기능이라 하려면 rebase+force push라는 위험한 예외 절차가 필요하고, 유일하게 가벼운 삭제는 아직 공유되지 않은 마지막 커밋의 reset뿐이다.
- ✨ "우리는 업로드가 곧 push라 로컬 단계가 없다"는 반론에는, push된 커밋을 못 지우는 이유가 push라는 행위 자체가 아니라 **다른 사람이 그 위에 작업을 쌓았을 가능성(공유)** 때문이라는 점을 짚어 반박할 수 있었다. "나만 쓰는 원격 저장소"인 상황에서는 push됐어도 최신 커밋 삭제가 안전하다 — 비유가 표면적으로 안 맞아 보여도 근거의 원천까지 파고들면 다시 살릴 수 있다.
