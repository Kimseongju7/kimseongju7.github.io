## 1일차: 머신러닝과 AI 인프라 기초

### 과정 개요
5일간 진행되는 과정으로, 앞의 3일은 "머신러닝 엔지니어링 온 AWS", 목·금 이틀은 "디벨로핑 제네릭 AI 애플리케이션(Developing Generative AI Applications)"을 다룬다. 테크니컬 인스트럭터 김현규 강사가 과정을 총괄하며, 강의는 45~50분 진행 후 10~15분 휴식하는 방식으로 운영된다. 참가자들은 컴퓨터공학·인공지능·통계학·자율전공 등 다양한 전공 출신이며, AWS 사용 경험은 처음인 인원과 학교 수업·프로젝트로 경험해본 인원이 절반씩이다. 대부분 파이썬 코드 작성·배포 경험이 있고, 업스테이지 LLM API를 활용한 추론 실습 경험이 있는 인원도 확인되었다. 검증·최적화에 대해 약 3분의 1은 모른다고 응답했고 나머지는 이해도가 있으나, 실제 구현 경험자는 1명뿐이었다. 반면 머신러닝·딥러닝·생성형 AI·에이전트 AI의 차이는 대체로 잘 이해하고 있었고, RAG 이해도는 높은 편이나 에이전트 워크플로우·시스템 이해도는 절반 정도였다.

교육 콘텐츠는 총 11개 모듈로 구성되며, 모듈 1(머신러닝 기초) → 모듈 2(비즈니스 문제의 머신러닝 매핑) → 모듈 3·4(데이터 준비) → 모듈 5·6(모델 훈련) → 모듈 7(모델 평가·튜닝) → 모델 배포·보안 → MLOps·모델 모니터링 순으로 진행된다. 진행 방식은 설문을 통해 "이론적 공백을 최소화하며 핵심 위주로 압축 진행하고, 수요일에 별도 워크숍/실습 세션을 추가"하는 방향으로 확정되었다. 1일차 목표는 모듈 5(가능하면 6)까지, 2일차까지 11개 모듈을 모두 마치는 것이다.

### 머신러닝과 AI 기초 개념
일반적인 프로그램은 개발자가 과거 데이터를 분석해 규칙을 도출하고 이를 코드로 구현하는 방식으로 동작한다. 반면 머신러닝은 과거 데이터와 알고리즘을 기계에 입력해 데이터 안의 패턴과 규칙을 기계가 스스로 학습하게 하고, 학습된 모델에 새 데이터를 넣어 미래 값을 예측한다. 인공지능 개념 자체는 1950년대부터 연구되었으며, 초기에는 인간이 모든 규칙을 직접 코딩하는 규칙 기반 시스템(예: 초기 자율주행)이 쓰였다.

이후 데이터와 컴퓨팅 파워가 늘면서 딥러닝 시대가 열려 비선형 데이터까지 추론할 수 있게 되었고(알파고 등), 트랜스포머·어텐션 메커니즘의 발전으로 콘텐츠를 직접 생성하는 생성형 AI 시대로 이어졌다. 딥러닝은 입력층·은닉층·출력층으로 구성되며, 각 은닉층·출력층에 활성화 함수를 적용해 비선형 문제를 해결한다는 점이 전통적 머신러닝과의 핵심 차이다. 컴퓨터 비전에서는 이미지 전체 범위에서 시작해 점차 모서리·라인 같은 세부 특징을 추출하는 방식으로 학습이 진행된다.

머신러닝 학습 방식은 지도학습·비지도학습·강화학습 세 가지로 나뉜다. 지도학습은 정답 라벨을 붙인 데이터로 기계를 학습시키는 방식(예: 이미지에 고양이/강아지 라벨 부착)이고, 비지도학습은 라벨 없는 대량 데이터에서 기계가 스스로 패턴을 찾는 방식(더 많은 데이터·자원·시간이 필요하며 결과 모델도 더 복잡)이다. 강화학습은 행동 결과에 보상/페널티를 부여해 보상을 최대화하는 방향으로 학습시키며, 자율주행이 대표 사례다.

지도학습은 다시 분류(범주를 나누는 작업)와 회귀(연료 소비량, 전기세, 칼로리, 임금 등 수치를 예측)로 나뉘며, 대표 알고리즘으로 그레이디언트 부스팅·XGBoost·의사결정나무·서포트 벡터 머신·최근접 이웃·선형 회귀 등이 있다. 비지도학습의 대표 기법은 데이터를 그룹화하는 클러스터링(K-means 등)과, 변수 간 관계를 파악해 추천에 활용하는 연관 분석(예: 김밥 구매 고객에게 김치 추천)이다. 차원 축소는 데이터의 차원(변수 개수)을 줄여 훈련 속도를 높이고 오차를 최소화하는 기법이다.

머신러닝 파이프라인 전체 흐름은 데이터 수집 → 탐색(이상치·노이즈 확인) → 전처리(정제·변환) → 훈련/검증/테스트 분할 → 알고리즘 선택 및 훈련 → 성능 평가 → 하이퍼파라미터 튜닝 → 목표 오차 도달 시 컨테이너 형태로 패키징 → 배포(엔드포인트 생성) → 배포 후 지속적인 성능 모니터링 및 재조정 순으로 이어진다. 애플리케이션은 프론트엔드·백엔드·데이터베이스로 구성되며, 사용자가 UI로 요청하면 백엔드가 모델에 전달하고 추론 결과를 받아 최종 응답을 제공한다.

### 머신러닝 프로젝트의 조직과 역할
- **데이터 과학자**: 데이터 수집·전처리·피처 엔지니어링·모델 구축 및 훈련을 담당한다.
- **데이터 엔지니어**: 데이터 과학자가 업무를 수행할 수 있도록 스파크·파이프라인 등 기반 인프라를 구축한다.
- **MLOps 엔지니어**: 모델 배포·운영·유지 관리를 담당하며 AWS CloudWatch 등으로 시스템을 모니터링하고, 쿠버네티스 등 배포 환경과 ML 워크플로우 지원 솔루션을 관리한다.
- **머신러닝 엔지니어**: 위 인프라·워크플로우를 포함한 머신러닝 시스템 전반을 총괄 관리한다.

### AWS AI 스택과 SageMaker
AWS의 AI 스택은 인프라·플랫폼·서비스 세 범주로 나뉜다. 인프라는 머신러닝·생성형 AI·에이전틱 AI 파이프라인 수행에 필요한 컴퓨팅·스토리지 자원(GPU 가상 서버, NVIDIA GPU나 AWS 자체 칩 인스턴스 등)을 제공한다. 플랫폼(SageMaker AI 등)은 모델 개발부터 배포, MLOps, 거버넌스, 데이터 전처리까지 엔드투엔드 개발 인프라를 지원한다. 서비스는 이미 모델까지 구축되어 즉시 도입 가능한 형태로, Amazon Translate·Polly·Transcribe(번역/음성), 아마존 커넥트(고객센터 AI), 퀵사이트 등이 있으며, 컴퓨터 비전·문서 이미지 추출·추천 엔진·사기 탐지 모델 등에 활용된다. 개발 여력이 부족할 때 이런 서비스형 모델을 조합해 시간과 비용을 절감할 수 있다(예: 클로드 코드 위스퍼러 사례).

SageMaker AI는 데이터 준비/전처리, 특성(피처) 중앙 관리, 노트북 등 개발 환경, 모델 훈련(사전 훈련된 모델·내장 알고리즘 지원으로 PyTorch/TensorFlow를 직접 구축할 필요 없음), 하이퍼팟(HyperPod) 클러스터, 오토ML(코드 없이 모델 훈련), 튜닝, 배치 추론, 모델 편향 감지/테스트, 데이터 랭글러, 디버깅 도구, 실험 관리, 데이터 레이블링까지 머신러닝 엔드투엔드 개발에 필요한 기능을 통합 제공한다. 자체적으로 머신러닝을 구현하면 프레임워크뿐 아니라 인프라·스토리지·파이프라인까지 직접 구축해야 하는 번거로움이 있다.

### Responsible AI
AI가 가짜 뉴스·딥페이크 생성(성범죄물 제작·유통 포함)에 악용되는 사례가 늘면서(예: 미국 대선 기간 GPT를 이용한 가짜 뉴스 생성 사례) 책임 있는 AI(Responsible AI)의 중요성이 커지고 있다. 기술 발전 속도에 비해 정책·법률 대응이 지연되는 가운데, 대한민국은 세계 최초로 AI 기본법을 제정·시행했으며 실질적 도입 시점은 EU AI 법안보다도 앞선다. AI 기본법은 AI 개발·서비스 제공 사업자에 대한 지원과 규제를 목적으로 하며, 의료·체육·대출·교통·공공 서비스·교육 분야는 "영향 AI"로 분류되어 더 엄격한 관리를 받는다. 콘텐츠 플랫폼에서는 AI 생성 콘텐츠임을 사용자가 즉시 식별할 수 있도록 표시하는 사례도 늘고 있다.

기존 애플리케이션은 조건문·반복문으로 결정론적 결과를 내지만, AI 모델은 확률 기반으로 콘텐츠를 생성·판단하므로 비결정론적이며 프롬프트 인젝션 등 새로운 유형의 보안 위협에 노출된다. 이 때문에 Responsible AI 도입이 필수적이며, 핵심 구성 요소로 공정성·설명 가능성·투명성·보안·진실성·거버넌스·안전성·제어 가능성이 꼽힌다. 설명 가능성은 모델이 특정 결정을 내린 이유를 명확히 설명할 수 있는 능력을 의미하는데, 변수 차원이 낮은 단순 모델은 추론이 쉽지만 수천억 파라미터의 LLM 등 딥러닝/생성형 AI 모델은 해석이 어렵다. 의학·법률처럼 의사결정이 중요한 분야일수록 Responsible AI 도입이 확대되고 있으며, 이는 데이터 부족 문제 해결과 모델 편향성 완화에도 기여한다.

### 머신러닝 비즈니스 과제 선정 기준
머신러닝 도입 여부는 다음 기준으로 판단한다: ①데이터 안에 패턴/관계가 존재하는지, ②충분한 데이터가 확보되어 있는지(일반적으로 특성 개수의 10배 이상 데이터 필요), ③지속적인 학습이 필요한 문제인지, ④성과를 수치로 정의할 수 있는지(예: 고객 이탈률 10% 감소, 사기 탐지율 25% 상승), ⑤일반 애플리케이션·기존 솔루션 대비 실제로 더 나은지. 간단한 알고리즘으로 해결 가능한 문제라면 굳이 머신러닝을 도입할 필요가 없으며, 머신러닝 모델 개발은 생성형 AI만큼은 아니어도 상당한 시간·비용이 든다.

다뤄진 사례:
- **이커머스**: 계정 도용·결제 사기·허위 제품 등록 문제 해결. 계정 손상률 감소와 허위 등록 제품 제거를 목표로, 신뢰도 개선과 손실 감소를 성공 지표로 설정(이 과제와 관련해 8,600이라는 수치가 언급됨).
- **마케팅**: 연령·위치·성별·직업 등 CSV 데이터로 소득 수준을 예측해 매출 증대. 소득과 상관관계가 낮은 성별·인종 등은 특성에서 제외 검토.
- **비즈니스 도서관**: 개인화 마케팅 콘텐츠 생성, 콘텐츠 제작, 실시간 감정 분석, 구매 패턴 예측 네 가지 과제 중 구매 패턴(구매 여부 예측) 분석이 머신러닝 적용 핵심 분야로 선정, 매출 목표 20% 달성 전략과 연계.
- 변화하는 트렌드(러닝, 트레일 러닝, 하이록스 등)에 맞춘 개인화 추천처럼 전통적 앱 개발보다 머신러닝 재훈련/튜닝이 효율적인 경우도 논의됨.
- 참고 학습 자료로 생활코딩의 머신러닝 과정이 소개됨(소비 패턴 분석 절약 기능, 해외 주식 변동성 분석, 이미지 유사 상품 추천 등 아이디어 사례 포함).

### 데이터 준비: 저장소와 형식
데이터베이스는 애플리케이션의 읽기/쓰기 트랜잭션(OLTP) 처리용 저장 공간이고, 데이터 웨어하우스·데이터 레이크는 분석·활용(OLAP) 목적의 저장 공간이다. 데이터 레이크는 문서·PDF 등 가공되지 않은 원시(로우) 데이터를 저장하며, 데이터 웨어하우스는 정제·변환된 데이터(정형 데이터 위주, 반정형 데이터도 일부 포함)를 저장한다. 데이터 마트(특정 목적용 별도 저장소)는 최근 실무에서 활용 빈도가 낮아지는 추세다. 지도학습(분류) 문제의 경우 고객 데이터는 보통 데이터베이스/데이터 웨어하우스에서, 컴퓨터 비전(제품 결함 판별용 사진 등)은 데이터 레이크에서 소스를 가져온다.

ETL(Extract-Transform-Load)은 정형·반정형 데이터를 추출·변환·적재하는 처리 과정이다. 데이터 품질의 핵심 기준은 정확성(오류 없음), 완전성(중요 변수 누락 없음), 일관성(단위·형식 통일, 예: 달러/원화/엔화 혼용 방지), 최신성(실시간 예측엔 최신 데이터 필요), 대표성(문제 해결에 유효한 특성만 선별)이다.

데이터 유형은 자연어(NLP: 감정분석·챗봇·문서요약), 테이블(분류·회귀), 시계열(주가·부동산 예측 등 타임스탬프 포함) 등으로 나뉘고, 정형/비정형/반정형으로도 분류된다. 저장 포맷은 크게 행 기반, 열 기반, 객체 표기법(JSON 등)으로 나뉜다. 행 기반 포맷은 데이터 수집·ETL 처리에 적합하고, 열 기반 포맷은 특정 열만 선택적으로 읽을 수 있어 모델 훈련·피처 엔지니어링에 유리하며 압축 효율도 더 높다(ORC가 Parquet보다 하이브 환경에서 효율적). Avro는 바이너리+스키마 구조의 직렬화 포맷이며, SageMaker의 RecordIO는 데이터를 레코드 단위 청크로 나눠 바이너리로 저장하는 훈련 데이터 포맷이다. 실무에서는 CSV 포맷이 여전히 많이 쓰이며, SageMaker 알고리즘별로 요구 포맷이 다르므로(예: 일반 분류는 CSV, 이미지 분류는 RecordIO/이미지 파일, NLP 시퀀스 모델링은 RecordIO, TensorFlow 텍스트 분류는 CSV/텍스트) 공식 문서에서 확인이 필요하다. 데이터 수집 방식도 주기적인 배치와 실시간 스트리밍(Kafka, Kinesis)으로 구분된다.

스토리지 옵션은 다음과 같이 선택한다: **S3**는 인터넷 기반 객체 스토리지로 무제한 저장이 가능해 데이터 레이크·모델 아티팩트 저장에 쓰이며 HTTPS로 GET/PUT 방식 입출력을 한다. **EBS**는 EC2에 연결하는 SSD/HDD 드라이브다. **EFS**는 NAS와 유사한 네트워크 공유 파일 스토리지로 여러 컴퓨팅 인스턴스 간 데이터셋·훈련 코드·모델 아티팩트 공유 및 분산 훈련에 적합하다. **FSx for Lustre**는 입출력 속도가 빠르고 지연이 짧아 대용량 데이터셋 훈련이나 실시간 예측에 적합하지만 장애 시 데이터 손실 위험이 있어 평소엔 S3에 데이터를 두고 훈련 시에만 Lustre로 옮겨 사용한다. SageMaker는 데이터셋이 EFS에 있으면 EFS를, 50~100GB 미만이면 S3 파일 모듈을, 50GB를 초과하면 개별 파일이 50MB 미만인지 등을 확인해 적절한 옵션을 선택하도록 제한한다.

### 데이터 탐색과 시각화
데이터 분석 시 볼륨·유형 파악 후 시각화를 통해 분포와 이상치를 확인한다. 범주형 데이터는 막대 차트·파이 차트·히트맵으로, 수치형(연속형) 데이터는 히스토그램·산점도로 분포를 비교한다. 히스토그램은 데이터가 좌우로 치우쳤는지, 이상치가 있는지 수치 단위로 보여주며, 바이올린 플롯은 그룹 간(예: 도시별 소득 수준) 분포 형태를 비교하는 데 유용하다.

평균은 이상치에 왜곡되기 쉬워 분위수(1/4, 1/2, 3/4 지점)를 함께 확인한다. 박스 플롯(상자 그림)에서 중앙선은 중앙값(50%), 상자 하단/상단 모서리는 각각 25%/75% 분위수를 나타내고, 상자 밖 수염(whisker)을 벗어난 값이 이상치다. 이상치는 근거가 있는 경우(예: 특정 지역의 소득 수준)도 있어 훈련 데이터 포함 여부를 신중히 판단해야 한다.

변수 간 관계는 산점도(scatter plot)나 다변량 통계로 파악하며, 상관관계 행렬과 열지도(heatmap)로 특성 간 연관 강도·방향성을 시각화한다(보라색=음의 상관관계, 녹색=양의 상관관계, 1에 가까울수록 강한 상관관계, 예: 몸무게-비만도 상관계수 0.7~0.8). 상관관계가 지나치게 높은 두 변수는 데이터 중복성과 오버피팅을 막기 위해 하나를 특성에서 제외하기도 한다.

### 데이터 클리닝과 피처 엔지니어링
데이터 클리닝은 값 변환, 형식 수정, 이상치·중복 항목 제외, 누락 데이터 보완 등을 포함한다. 결측치 유형은 세 가지다: **완전 무작위 누락**(다른 변수와 무관하게 발생, 전산 오류 등 관련성 없는 경우 행/열 삭제로 처리), **비무작위 누락**(특정 집단이 응답을 회피해 발생, 가장 해결하기 어려움 — 예: 고소득층의 소득 응답 거부), **편향 유발 누락**(예: 특정 성별이 몸무게 기재를 꺼리는 경향). 처리 방법은 성별처럼 편향 유발 변수를 제거하거나, 키·나이 등 다른 변수로 값을 추정/대체하거나, 전문가 지식이나 유사 직군·지역 데이터를 참조해 값을 생성하는 것이다.

피처 엔지니어링은 원시 데이터를 계산·가공해 훈련에 의미 있는 특성(변수)을 만드는 과정이다. 범주형 데이터는 순서 유무에 따라 인코딩 방식이 달라진다.
- **레이블 인코딩**: 범주에 0부터 순차적인 숫자를 부여. 숫자 크기·순서가 잘못된 영향을 줄 수 있다는 한계가 있다.
- **원-핫 인코딩**: N개 범주를 N개의 비트 벡터(칼럼)로 분리해 해당 시 1, 아니면 0으로 표시. 선형 회귀·신경망처럼 수치 크기가 성능에 영향을 주는 모델에서 오인을 방지하지만, 범주가 늘수록 차원이 증가해 "차원의 저주" 문제가 생긴다.
- **순서형 인코딩**: 학력처럼 순서가 있는 데이터에 스몰(0)·미디움(1)·라지(2)처럼 순차적 수치를 부여해 순서 정보를 보존한다.
- **바이너리 인코딩**: 범주를 이진 코드로 변환(예: 레드/블루/그린/옐로우 4개 범주를 0과 1 조합으로 표현). 범주가 많을 때 오버랩으로 학습이 가능하다.

수치형 데이터는 스케일링으로 처리한다. **정규화**는 값을 0~1 또는 -1~1 범위로 제한하며 신경망·거리 기반 알고리즘에서 이상치 영향을 줄인다. **표준화**는 평균 0, 표준편차 1로 조정해 정규분포에 가깝게 만들며 선형/로지스틱 회귀 등 선형 모델에 적합하다(정규분포가 아닌 데이터에 쓰면 왜곡 가능). **비닝**은 연속형 데이터를 구간별 범주로 나눠(예: 점수 구간을 등급으로) 훈련 시간을 줄이지만 정보 손실이 있을 수 있다. 로그 변환 등으로 데이터 범위를 축소하기도 한다.

차원 축소는 상관관계가 낮은 특성을 제거하거나(예: 부동산 가격 예측에서 위치·등록일 등 4개 특성 중 상관 낮은 특성 제거로 4차원→2차원), **주성분 분석(PCA)**처럼 변수 간 연관 관계를 분석해 정보 손실을 최소화하며 차원을 줄이는 방식이 있다. PCA는 데이터 평균을 원점으로 이동(시프팅)한 뒤, 원점을 지나는 직선까지의 수선의 발 거리 제곱합(Sum of Square)이 최대가 되는 직선(PC1)을 찾고, PC1 기준으로 데이터를 통합해 차원을 줄인다(이 과정에서 정보 손실은 불가피). 예를 들어 주택 가격 예측 시 면적·침실·욕실 등 변수를 "크기"라는 하나의 차원으로, 가격·세율·주별 세금 체계 등 6개 변수를 2개 차원으로 통합할 수 있다.

이런 전처리는 SageMaker Data Wrangler(GUI 클릭만으로 전처리, 커스텀 코드 추가 가능)로 코딩 없이 수행하거나, SageMaker Processing(관리형 스크립트 실행) 또는 Amazon EMR(오픈소스 기반 대규모 ETL·빅데이터 처리, 하둡 에코시스템 활용 시)로 처리할 수 있다.

### 모델 훈련 기초
SageMaker에서 모델을 구현하는 방법은 크게 네 가지다: ①SageMaker 내장 알고리즘이 포함된 도커 이미지 사용(가장 간단), ②TensorFlow/PyTorch 등 프레임워크가 포함된 컨테이너에 사용자 스크립트만 추가, ③완전한 커스텀 컨테이너 직접 구성, ④마켓플레이스의 서드파티 알고리즘 사용. 알고리즘 선택 시 문제 유형(분류/회귀/클러스터링 등), 요구 데이터 포맷, 필요한 CPU/GPU 리소스와 소요 시간, 전통적 머신러닝 vs 딥러닝 여부를 함께 고려해야 한다. 정형 데이터 기반 지도학습(CSV 등)은 보통 CPU로 충분하지만, 비정형 데이터(네트워크·트랜스포머 등)는 GPU·분산 컴퓨팅이 필요해 비용이 커진다. SageMaker는 사용량 기반 과금 구조상 비용 부담이 크다는 지적이 많아, 세이빙스 플랜과 스팟 인스턴스로 비용을 절감할 수 있다.

지도학습 알고리즘(그레이디언트 부스팅, XGBoost, KNN, 선형 학습자, 결정 나무, SVM 등)과 비지도학습 알고리즘(랜덤 포레스트, IP 인사이트, K-평균, PCA, 이상 탐지 등), 텍스트/음성용 시퀀스 모델·토픽 모델링 등 다양한 카테고리의 내장 알고리즘이 제공된다. 임베딩을 통해 데이터를 벡터화해 RAG 시스템 등에 활용할 수도 있다.

**XGBoost 동작 원리**: 1차 훈련 모델의 예측값과 실제 레이블 간 오차를 계산하고, 이 오차를 줄이는 방향으로 트리를 반복 추가해 최종 모델을 만드는 부스팅 알고리즘이다. Decision Tree를 개선한 방식으로, 데이터를 분할할 때마다 유선(gain) 점수를 계산한다. 예시: 정규화 파라미터 알파를 0으로 두면 오차 제곱합을 오차 개수로 나눈 유선 점수가 16으로 계산되고, 데이터를 좌/중/우로 분할해 각 영역의 유선 점수를 비교했을 때 가장 오른쪽 영역(전체의 약 71%)에서 가장 높은 이득이 나와 해당 방향으로 분할, 이를 기반으로 두 번째 모델을 만들어 첫 번째 모델의 오차를 보정한다. 이 과정을 더 이상 이득을 얻을 수 없거나 데이터가 소진될 때까지 반복한다.

**경사하강법**은 손실 함수의 기울기가 0이 될 때까지 가중치를 반복 업데이트하는 최적화 기법이다. 훈련 데이터셋을 한 번 모두 학습하는 단위를 에포크라 하며, 방식은 세 가지다: **배치 경사하강법**(전체 데이터로 1에포크 학습 후 가중치 업데이트 — 정확하지만 느림), **확률적 경사하강법**(샘플 하나마다 업데이트, 예: 1,000개 데이터셋이면 1,000번 업데이트 — 빠르지만 노이즈에 취약), **미니배치 경사하강법**(데이터를 일정 크기 배치로 나눠 배치 완료 시마다 업데이트, 예: 1,000개를 100개씩 10배치로 나누면 10번 배치 완료가 1에포크 — 둘의 장단점을 절충).

손실 함수는 예측값과 실제값 차이를 계산하는 지표로, 회귀 모델은 평균 제곱근 오차(RMSE) 등을, 로지스틱 회귀 등 분류 모델은 로그 가능도(log-likelihood)를 사용한다. 로지스틱 회귀는 특정 기준 점수로 결과를 0 또는 1로 분류하는 모델이며, 최적화에는 경사하강법을 쓴다.

**과대적합/과소적합**: 과소적합은 모델이 훈련 데이터의 패턴을 충분히 포착하지 못한 상태다. 과적합은 훈련 데이터에 과도하게 튜닝되어 노이즈까지 학습해 일반화 성능(검증 데이터 성능)이 떨어지는 상태로, 훈련 데이터의 다양성/볼륨을 늘리거나 정규화, 모델 복잡도 감소로 해결한다. 과소적합은 반대로 모델 복잡도를 높여 개선한다. 이는 편향-분산 트레이드오프로도 설명되는데, 과적합은 편향은 낮지만 분산이 높은 상태, 과소적합은 분산은 낮지만 편향이 높은 상태이며 최적 모델은 이 둘의 균형점을 찾는다.

데이터 분할 비율은 1일차에는 훈련용 60~70% / 검증용 10~20% / 테스트용 10% 기준이, 2일차에는 6:2:2, 7:2:1, 8:1:1(훈련:검증:테스트) 등의 비율이 언급되었다. 훈련 데이터로 모델을 학습시키고, 검증 데이터로 성능을 평가하며 하이퍼파라미터를 튜닝하고, 최종적으로 테스트 데이터로 배포 전 마지막 평가를 수행한다. 훈련은 알고리즘과 데이터를 결합해 모델이 특성별 가중치를 스스로 학습하는 과정이다.

실무에서는 SageMaker Python SDK로 Estimator 객체를 선언(권한 Role, 인스턴스 개수·유형 지정)하고 image_uri 모듈로 원하는 버전의 알고리즘(예: XGBoost) 컨테이너 이미지를 불러온 뒤, S3의 훈련/검증 데이터 경로를 지정해 훈련(fit)을 실행한다. 훈련 작업은 AWS 관리형 EC2 클러스터 인스턴스에서 컨테이너 기반으로 실행되며, 완료된 모델(가중치·성능 지표 포함)은 S3에 저장된다. 코딩이 부담스러운 사용자를 위해 **SageMaker Autopilot**(오토ML 기반 노코드 모델 구축, 이미지/텍스트/테이블 등 다양한 데이터 지원)과 **SageMaker Canvas**(코딩 없는 모델링, 퀵 모드는 프로토타입용으로 데이터셋이 작으면 약 30분, 스탠다드 모드는 프로덕션용으로 2~5시간 소요)도 제공된다.

모델 선택 시 해석 가능성과 비용의 트레이드오프를 고려해야 한다. 교육·의료처럼 고위험 영역은 해석 가능성이 필수적이며, 단순 선형 회귀는 해석이 쉽지만 SVM·신경망 등 복잡한 모델은 해석이 어려운 대신 비선형 문제 예측 정확도가 높다.

### 1일차 실습 안내
- **실습 1**: SageMaker Data Wrangler(SageMaker Canvas 통해 접속)를 활용한 GUI 기반 데이터 전처리, 또는 EMR 클러스터 + Spark 엔진을 활용한 데이터 전처리.
- **실습 2**: SageMaker Processing과 제공된 파이썬 코드로 데이터 전처리. 실습 1번은 약 1시간 15분, 실습 1·2번을 합친 전체 실습 시간은 쉬는 시간 포함 1시간 30분 내외였으며(실습 2번 소요 시간이 길어 1·2번을 병행 진행), 이후 실습 1·2번을 통합해 총 1시간 40분으로 배정된 회차도 있었다.

- 1일차 목표는 총 11개의 노트를 완료하는 것이었고, 다음 날 실습을 위해 당일 교육 내용을 바탕으로 노트를 작성할 예정이었으며 새로 합류한 교육생에게는 별도 과제가 부여됐다. 2일차 후반부 기준 남은 교육 과정은 이론 8회·실습 4회, 총 6시간 정도로 예상됐다.

## 2일차: 모델 평가, 배포, 보안, MLOps

### 하이퍼파라미터와 모델 훈련 환경
하이퍼파라미터는 모델·알고리즘마다 설정값이 다르며, 학습률·에포크·미니배치 크기 등 공통 하이퍼파라미터는 실습 훈련 코드 실행 시 직접 확인한다. 일반적인 머신러닝·전처리·피처 엔지니어링에는 CPU 인스턴스를, 딥러닝·AI 모델 개발에는 GPU 인스턴스를 사용한다. **모델 레지스트리**는 모델 그룹을 만들어 여러 버전을 중앙에서 관리하며, 조직 차원의 모델 거버넌스(정책·프로세스 통합 관리)를 구현하는 데 쓰인다.

### 모델 평가 지표
분류 모델 평가는 **혼동 행렬(Confusion Matrix)**에서 시작한다. 참(True)은 예측과 실제 값이 일치하는 경우, 거짓(False)은 불일치하는 경우다. **정확도(Accuracy)**는 전체 중 올바르게 예측한 비율이지만, 데이터 불균형이 심하면(예: 전체 2,470개 샘플 중 긍정이 1,620개) 왜곡될 수 있어 다른 지표와 함께 봐야 한다. **정밀도(Precision)**는 긍정으로 예측한 것 중 실제 긍정 비율로, 스팸 메일 분류처럼 긍정 예측(거짓 긍정, False Positive 최소화)이 중요한 경우에 활용한다. **재현율(Recall)**은 암 진단처럼 거짓 부정(False Negative)을 최소화해야 하는 경우에 중요하다. **F1-Score**는 정밀도와 재현율의 조화 평균으로, 산업 현장에서 종합 성능 지표로 널리 쓰인다.

이진 분류 모델은 참/거짓을 직접 출력하는 게 아니라 참일 확률을 계산하며, 이 확률에 임계값을 설정해 참/거짓으로 최종 분류한다. 임계값이 낮아질수록 참 긍정 비율(TPR)과 거짓 긍정 비율(FPR)이 함께 상승한다. **ROC 곡선**은 FPR(x축)과 TPR(y축)로 임계값 변화에 따른 성능을 시각화하며, 곡선이 좌측 상단에 가까울수록 좋은 모델이다. **AUC**는 ROC 곡선 아래 면적으로, 값이 클수록(1에 가까울수록) 모델 성능이 우수함을 의미한다.

### 하이퍼파라미터 튜닝과 분산 훈련
알고리즘 튜닝 기법으로 **그리드 서치**(가능한 모든 조합을 학습), **랜덤 서치**, **베이지안 최적화**(이전 결과를 바탕으로 다음 값을 예측), **하이퍼밴드**(중간 체크포인트로 성능 좋은 세트에 자원 집중, 예: 80개 하이퍼파라미터 세트 중 우수한 세트에 자원 집중)가 있다. 훈련 횟수가 늘수록 시간·비용도 비례해 늘어나므로 효율적인 자원 투자가 필요하다. 튜닝 성능은 검증 데이터의 AUC 값으로 측정한다.

**분산 훈련**은 메모리·컴퓨팅 리소스를 분산해 훈련 시간을 단축하며, 모델 병렬화(가중치를 분할해 훈련 후 결합), 데이터 병렬화(대용량 데이터를 나눠 학습), 하이브리드(둘의 결합) 방식이 있다. SageMaker의 **HyperPod**는 컴퓨팅 클러스터를 생성해 분산 처리를 지원하며, 수십억 파라미터·대용량 이미지 데이터 처리 시 훈련 시간을 단축한다.

### 모델 배포 전략과 추론 방식
빌드된 모델 아티팩트는 S3에 저장되고, 이를 참조하는 SageMaker 엔드포인트를 생성해 클라이언트가 API(보통 API 게이트웨이 경유)로 추론을 요청한다. 배포 전략은 일반 애플리케이션 배포 전략과 크게 다르지 않다.
- **인플레이스/롤링 배포**: 기존 환경 내에서 모델을 교체하며, 롤링은 여러 인스턴스를 하나씩 순차적으로 교체하는 방식이다. 속도가 빠르고 비용 효율적이지만 문제 발생 시 롤백이 번거롭다.
- **블루그린 배포**: 기존 환경(블루)과 새 환경(그린)을 동시에 운영하다 라우팅을 전환한다. 문제 발생 시 이전 버전으로 즉시 되돌릴 수 있어 안전하지만, 환경을 미리 구축해야 해 상대적으로 느리고 환경 격리 수준이 높다.
- **선형(카나리) 배포**: 트래픽을 일부만 신규 환경으로 보내 검증 후 점진적으로 확대하는 방식으로(예: 5분마다 20%씩 전환), 가장 안전하지만 배포 속도가 느리다.

추론 방식은 네 가지다.
- **실시간 추론**: EC2 기반 엔드포인트를 상시 구성해 요청에 즉시 응답. 24시간 상시 실행.
- **서버리스 추론**: AWS Lambda(런타임+라이브러리+코드로 구성된 함수 단위) 기반. 평소엔 비활성 상태이다가 요청 시 마이크로 VM 환경을 생성해 처리하며, 이 지연을 **콜드 스타트**라 한다(15분간 요청이 없으면 환경 자동 소멸). 인프라 관리 부담이 없지만 콜드 스타트 지연이 있으며 CPU 기반 추론만 지원한다.
- **비동기식 추론**: 요청을 대기열에 저장해 처리, 요청량이 많거나 처리 시간이 긴 작업에 적합.
- **배치 변환**: 실시간 응답이 필요 없는 데이터 분석·인사이트 도출·관리적 목적의 대량 추론에 사용, 비용 절감 효과가 있다.

추론 인프라 구축 시 EKS·ECS·Lambda 등 AWS 컴퓨팅 서비스 중 선택하며, 관리형 서비스(SageMaker 엔드포인트)는 운영 편의성이 높은 대신 커스터마이징이 제한적이고, EKS(쿠버네티스)는 서드파티 도구 연동 등 커스터마이징 자유도가 높은 대신 관리 포인트가 늘어나는 트레이드오프가 있다.

**엔드포인트 생성 절차**는 ①S3 모델을 컨테이너 이미지와 연결해 모델 객체 생성 → ②`create_endpoint_config`로 엔드포인트 구성(구성 이름, 변수명과 참조 모델, 인스턴스 유형 지정) → ③엔드포인트 배포의 3단계로 진행된다. 구성 요소에는 리소스 오토스케일링 정책(클라이언트 호출량에 따라 자동 조절), 데이터 캡처, 로깅, 보안 제어, 배포 정책이 포함된다.

**다중 모델 엔드포인트**는 동일 프레임워크 기반의 여러 모델을 하나의 엔드포인트/인스턴스에서 운영해 자원 활용도를 높이며(A/B 테스트, 목적별 분리, 멀티 테넌트에 활용), 호출 시 API 파라미터에 엔드포인트 이름과 타겟 모델을 함께 명시해야 한다. **다중 컨테이너 엔드포인트**는 서로 다른 프레임워크로 전처리·추론·후처리를 하나의 엔드포인트에서 파이프라인처럼 순차 실행하며, 호출 시 타겟 컨테이너 호스트 네임을 지정한다. **Amazon SageMaker Inference Recommender**는 인스턴스 수량·컨테이너 파라미터·메모리 크기 등을 분석해 최적의 인스턴스 유형·구성을 추천, 성능과 비용을 동시에 최적화한다.

프로덕션 환경에서는 비용·성능보다 안전성·가용성을 우선해야 하며(서버리스보다 예측 가능한 안정적 구성 권장), 문제 발생 시 대응을 위해 자동 롤백 기능을 구성하는 것이 중요하다.

### 2일차 실습(실습 3~6)
- **실습 3**: 소득 데이터를 이용해 연봉 5만 달러 초과 여부를 예측하는 XGBoost 이진 분류 모델을 훈련. SageMaker Debugger로 성능 보고서를 생성해 레이블 분포·손실 비교 그래프·특성 중요도로 과적합 여부를 점검(훈련·검증 오차 격차 증가 시 과적합 의심, 훈련 시 FP32 정밀도 활용).
- **실습 4(랩4)**: XGBoost 하이퍼파라미터 튜닝 실습. 코드에서 하이퍼파라미터 레인지를 설정하면 SageMaker가 범위 내에서 자동으로 조합을 생성해 튜닝하며, 검증 데이터 AUC로 성능을 비교한다.
- **회귀 모델 실습**: 아이템 ID·상점 ID·시간대별 구매 정보·제품 가격·스코어 등이 담긴 CSV 데이터(2017년 10월 1일~2019년 12월 타임스탬프)로, 처음엔 향후 5년간 제품별 가격을 예측하려 했으나 예측 대상이 가격이 아닌 수요임을 정정해 향후 5개월간 아이템별 수요를 예측하는 회귀 모델을 구축했다. 학습 데이터는 20명 대상 27개월치 시계열 데이터로 단기 시계열에 최적화된 알고리즘을 사용했고, RMSE(Root Mean Square Error) 최소화가 목표였다. 모델 빌드 유형은 예측·테스트용 퀵 빌드(약 30분, 프로토타입), 커스터마이징이 필요한 커스터마이즈 빌드, 약 3~4시간 후 코딩·GUI 작업이 가능해지는 스탠더드 빌드로 구분됐다.
- **실습 5**: 블루그린 배포 + CloudWatch 기반 자동 롤백 실습. 서로 다른 특징(그중 하나는 의도적 오류 포함)을 가진 3개 모델을 준비해 배포 후 장애 상황을 시뮬레이션한다. 엔드포인트 구성을 모델 1→모델 2로 교체하며 트래픽을 20%씩 선형적으로 전환하고 각 단계마다 1분간 대기한다. CloudWatch로 SageMaker 엔드포인트 변형 지표를 60초 간격으로 수집하고, `Auto Rollback Alarm` 함수로 경고를 정의한다. API 500 에러가 60초 동안 1회 이상 발생해 1% 임계치를 초과하거나, 모델 레이턴시가 10ms 이상 지속되면 경고 상태로 전환되어 자동 롤백이 트리거된다.
- **실습 6**: 실무에서 API와 AI 기술을 프로젝트에 적용하는 방법을 익히는 실습(약 1시간 소요).

### 보안과 거버넌스
계정·인프라 보호의 핵심 요소는 인증과 권한, 네트워크, 인프라 구성, CI/CD 파이프라인 보안이다. AWS 계정 인증·권한 관리는 **IAM**으로 수행하며, 계정 이메일·비밀번호를 다른 서비스와 동일하게 쓰는 것은 보안상 위험하다. **MFA**를 설정하면 외부에서 탈취한 계정 정보만으로는 로그인할 수 없어 계정 보안이 크게 강화된다(해커가 로그인에 실패하면 코인 채굴 등 다른 방식의 공격을 시도할 가능성도 언급됨). 루트 사용자 보호가 최우선이며, 개발자·운영자·데이터 사이언티스트 등 역할별로 개별 IAM 사용자를 만들어 필요한 최소 권한만 부여해야 한다.

IAM 인증 방식은 관리 콘솔 로그인(사용자 이름·비밀번호)과 API 호출용 액세스 키(액세스 키 ID + 시크릿 액세스 키) 두 가지다. SDK 작업은 액세스 키만으로 가능하며, 콘솔 작업이 필요한 경우에만 별도 콘솔 접근 권한을 부여한다. 실습 환경은 AWS 빌더 ID 기반 SSO(싱글 사인온)로 자동 로그인되는 구조다. 권한 정책은 아이덴티티 기반 정책(사용자/역할에 부여)과 리소스 기반 정책(S3 버킷, API 게이트웨이 등 리소스에 직접 적용)으로 구분된다. 데이터 사이언티스트·MLOps 엔지니어·데이터 엔지니어·머신러닝 엔지니어 등 직무별로 SageMaker AI 사용 권한을 적절히 부여해야 한다.

SageMaker Studio는 기본적으로 인터넷을 통해 퍼블릭하게 통신하는데, 엔터프라이즈 환경에서는 보안을 위해 **VPC 온리 모드**로 설정해 특정 VPC·서브넷을 통해서만 트래픽이 흐르도록 제한하고, 프라이빗 링크로 VPC 내 통신을 구성한다. 이 경우 외부 패키지를 직접 수신할 수 없어 커스터머 VPC 내에 별도 게이트웨이를 구현하고 인터넷 게이트웨이와 연결해야 외부 통신이 가능하다. 네트워크 보안은 보안 그룹(인스턴스 단위 방화벽), 네트워크 ACL(서브넷 단위 방화벽), 네트워크 방화벽 세 가지를 함께 활용하며, 서브넷별 트래픽 흐름은 라우팅으로 제어한다. VPC Flow Logs로 트래픽을 캡처·분석할 수 있다.

데이터 암호화는 **Amazon KMS**의 암호화 키로 학습 데이터·애플리케이션 데이터·테스트 데이터·모델 아키텍처·공유 코드 등 파이프라인 전 구간에 적용해야 하며, 동일 키가 있어야만 복호화할 수 있다. 그 외에도 CloudWatch(지표 로그 수집·분석), Config(S3 암호화 여부 등 규정 준수 자동 검사), API 호출/응답 로그, EC2 취약점 탐지, SSH 무차별 대입 공격 등 지능형 공격 실시간 탐지 등의 보안 모니터링 체계가 필요하다.

### MLOps와 CI/CD 파이프라인
**CI/CD 파이프라인**은 애플리케이션의 개발-테스트-빌드-검증-모니터링을 자동화하는 체계로, 커밋 이후 빌드→테스트→배포→모니터링 순으로 진행된다. **MLOps**는 이 소프트웨어 파이프라인 개념을 머신러닝에 맞게 자동화한 것으로, 개발자·운영자·데이터 작업자 간 긴밀한 협업 문화와 방법론에 머신러닝 기술을 결합한 개념이다. 데이터 파이프라인(데이터 관리·전처리)과 머신러닝 파이프라인(훈련·평가, 기준 충족 시 엔드포인트 생성까지 자동화)을 하나로 연결해 구현한다. 파이프라인 자체 보안(리포지토리 커밋 시 코드 분석·검증 자동화, 빌드 이미지의 모듈/패키지 취약점 검사, 패키지 보안 테스트, 최근엔 AI를 활용한 버전별 테스트도 증가)과 파이프라인 내부 보안을 구분해 체계적으로 구현해야 한다.

CI/CD 구현 예시: 모델 훈련·전처리용 코드와 데이터는 각각의 리포지토리에서 관리되며, 훈련 코드가 업데이트/커밋되면 데이터 검증-전처리-모델 학습-모델 검증을 포함한 파이프라인이 자동 실행된다. 최종 생성된 모델 파일은 애플리케이션 파이프라인의 업데이트를 트리거하고, 애플리케이션 코드·모델 파일 변경이 CI/CD 파이프라인을 자동 실행시켜 배포-검증-승인까지 이어진다. 전처리 코드 변경 시에는 별도의 파이프라인 검증용 테스트 코드로 문제 여부를 확인한다. **CodeSuite**(Jenkins와 유사한 빌드 서비스, buildspec YAML로 테스트 스크립트 설정)로 빌드를 자동화하고, SageMaker Inference Recommender에 대한 로드 테스트(오버시스 기반 캐시 패키지로 부하 테스트 효율화)를 수행한 뒤 세그먼트를 생성해 A/B 테스트를 진행하기도 한다.

**SageMaker Pipelines**는 데이터 처리·모델 훈련·파인튜닝·평가·모델 등록·배포 과정을 자동화하며, GUI 또는 SDK로 생성할 수 있고 둘 다 최종적으로는 JSON 스키마 형태로 정의된다. JSON 스키마는 조건부 분기(예: F1 스코어 등 성능 지표가 목표치 도달 여부 판단)를 지원하며, 모델 등록·튜닝·EMR·오토ML·엔드포인트 관련 작업까지 통합 구성할 수 있다. 코드로 등록한 파이프라인은 SageMaker Studio GUI에서 프로세싱·트레이닝·조건 설정·모델 등록 과정을 실시간으로 확인할 수 있다. **SageMaker 파이프라인**이 모델 훈련·빌드에 집중한다면, **SageMaker 프로젝트**는 이 파이프라인과 CI/CD 파이프라인을 통합해 전체 MLOps 구조를 구현한다.

전체 자동화 흐름의 예: **CodeCommit**(코드 버전 관리) → **CodeBuild**(프로세싱 코드 포함 빌드, SageMaker 파이프라인을 가동하는 API 호출 코드 실행) → 파이프라인이 모델 훈련을 자동 진행 → 모델이 레지스트리에 등록되며 상태가 변경되면 **EventBridge**가 이를 감지 → 설정된 규칙에 따라 코드 파이프라인을 자동으로 트리거해 이후 배포 단계로 이어진다.
