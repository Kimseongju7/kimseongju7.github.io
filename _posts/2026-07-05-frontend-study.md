프론트엔드 개발을 공부하면서 정리한 노트다. 프론트엔드가 하는 일과 개발 흐름부터 CSR·SSR 같은 렌더링 방식, React·Next.js·Node.js, 그리고 타입스크립트·Tailwind·패키지 매니저·테스트 같은 도구까지 훑는다.

## 프론트엔드가 하는 일

프론트엔드는 사람과 컴퓨터 사이의 통신, 즉 **양방향 번역**을 담당한다. 서버가 준 데이터를 사람이 읽을 수 있는 형태로 바꿔 보여주고, 반대로 클릭·타이핑·드래그 같은 사람의 물리적 행동을 감지해 그 의도를 백엔드가 알아들을 수 있는 형태(= API 요청)로 바꿔 전송한다.

그래서 프론트엔드만의 고유한 관심사가 생긴다.

- **상태(state) 관리** — 백엔드는 요청 하나 처리하고 끝나지만, 프론트엔드는 사용자가 화면에 머무는 동안 "지금 이 화면이 어떤 상태인가"를 계속 들고 있어야 한다 (열린 모달, 선택된 탭, 입력 중인 값 등).
- **즉각성(반응 속도)** — 사람은 클릭했는데 0.5초라도 반응이 없으면 "느리다"고 느낀다. 그래서 서버 응답을 기다리지 않고 먼저 화면을 바꾸는 기법(낙관적 업데이트)도 프론트엔드만의 고민이다.
- **다양한 환경 대응** — 같은 데이터라도 모바일/데스크톱, 느린 네트워크, 접근성이 필요한 사용자(스크린리더 등)처럼 조건이 제각각이라 그에 맞춰야 한다.

비유하자면, 백엔드가 "공장(생산·저장·계산)"이라면 프론트엔드는 "매장의 진열대와 점원"이다. 공장에서 만든 물건(데이터)을 손님이 고르기 쉽게 진열하고(렌더링), 손님이 원하는 걸 말하면(사용자 입력) 그걸 다시 공장에 전달할 주문서로 바꿔준다(API 요청). 물건 자체를 만들거나 재고를 계산하는 건 프론트엔드의 일이 아니다 — 그건 여전히 공장(백엔드)의 일이다.

## 개발 과정과 작업 사이클

프론트 개발의 큰 흐름은 다음과 같다.

> 화면 디자인 → 컴포넌트 쪼개기 → 컴포넌트 만들기 → 동작 구현 → 백엔드와 연결

구현하는 것은 결국 세 가지다: **HTML(뼈대) + CSS(생김새) + JS(동작)**. 이 셋을 프레임워크를 사용해 개발한다. 화면은 **[컴포넌트](/posts/component/)**(재사용 가능한 조각)를 조합하는 방식으로 만든다.

실제 작업 사이클을 단계로 풀면:

1. **디자인/스토리보드 확인** — 어떤 화면이 필요한지, 어떤 상태(로딩/에러/빈 화면)가 있는지 파악
2. **컴포넌트 단위로 쪼개기** — 큰 화면을 작은 조각(헤더, 리스트, 카드 등)으로 분해
3. **로컬 서버 띄우고 눈으로 확인** — `npm run dev` 같은 명령으로 브라우저에서 바로 확인하며 개발 (핫 리로드로 저장하면 바로 반영됨)
4. **상태 관리** — 화면에 보여줄 데이터를 어떻게 들고 있을지 (버튼 클릭 상태, 서버에서 받은 데이터 등)
5. **API 연동** — 백엔드(`backend/` 디렉터리)가 제공하는 API를 호출해 실제 데이터를 붙임
6. **스타일링** — CSS, 또는 Tailwind 같은 유틸리티 CSS 프레임워크로 디자인 적용
7. **테스트** — 컴포넌트가 의도대로 동작하는지 자동화 테스트 작성
8. **빌드 & 배포** — 개발용 코드를 최적화된 정적 파일로 빌드해 서버에 올림

## 렌더링 방식: CSR·SSR·SPA·MPA

**[CSR](/posts/csr/) (클라이언트 사이드 렌더링)** — 브라우저가 거의 빈 HTML을 받고, JS가 실행되면서 화면을 구성한다. 한 번 HTML과 JS를 받으면 이후 상호작용에서는 서버가 JSON 데이터만 내려주고, 사용자의 컴퓨터에서 그 데이터를 HTML로 조합해 렌더링한다. 즉 **조합은 사용자 쪽에서, 데이터는 서버가** 담당한다.

- 장점: 처음 로딩 후에는 페이지 전환이 매우 빠르다(새로고침 없이 화면만 바뀜). 서버가 하는 연산량이 확연히 줄고, 오가는 데이터 크기도 작아져 서버 부하가 적다.
- 단점: 첫 화면이 뜨기까지 매우 느리다. 첫 요청 시 HTML 파일보다 훨씬 큰 JS 번들을 가져와야 하기 때문이다. 크롤러가 빈 화면만 볼 수 있어 [SEO](/posts/seo/)에도 불리하다.

**[SSR](/posts/ssr/) (서버 사이드 렌더링)** — 서버가 이미 완성된 HTML을 만들어서 보내주고, 이후 JS가 붙어 상호작용한다. 요청을 받을 때마다 서버가 매번 HTML을 새로 렌더링해 브라우저로 전송한다(렌더링된 HTML 자체도 용량이 큼).

- 단점: 페이지 이동마다 서버 요청이 필요하다.

정리하면, 블로그나 랜딩 페이지처럼 SEO가 중요한 곳은 SSR이 좋고, 대시보드처럼 SEO가 중요하지 않은 내부 화면은 CSR로 충분하다. **Next.js는 페이지별로 둘을 섞어 쓸 수 있다.** (이 둘 말고 다른 방식도 존재한다.)

**[SPA](/posts/spa/) (Single Page Application)** — HTML 하나만 로드하고, 이후 페이지 이동은 실제 새로고침 없이 JS가 화면 내용만 바꿔치기하는 방식. CSR이 대부분 이런 식으로 동작한다.

**[MPA](/posts/mpa/)와의 차이** — MPA는 요청할 때마다 현재 페이지를 버리고 새로 렌더링한다. 반면 SPA는 링크 클릭을 가로채서 필요한 데이터만 요청한 뒤, 일부 [DOM](/posts/dom/)만 갈아끼운다.

## SSR을 담당하는 건 프론트엔드 서버다

혼동하기 쉬운 지점인데, SSR은 기존 백엔드(Java)가 HTML을 만든다는 뜻이 **아니다**. Next.js 같은 프레임워크를 쓰면, 프론트엔드 프레임워크 자체가 자기만의 서버(Node.js 기반)를 하나 더 갖게 된다.

```
[브라우저] → [Next.js 서버 (Node.js)] → [Java/Spring API 서버] → [DB]
              ↑ 여기서 HTML을 조립해서       ↑ 데이터만 JSON으로 응답
                브라우저로 보냄 (SSR)
```

즉 서버가 두 종류로 나뉜다.

1. **Java/Spring 백엔드** — 지금 그대로, 순수 데이터/비즈니스 로직 API 서버 (변화 없음)
2. **Next.js 프론트엔드 서버** — 사용자 요청이 오면 그 안에서 Java API를 호출해 데이터를 받아온 뒤, 그 데이터로 HTML을 조립해 브라우저에 완성된 페이지를 보냄

랜딩 페이지 요청이 들어올 때의 순서:

1. 브라우저가 Next.js 서버에 랜딩 페이지 요청
2. Next.js 서버가 (서버 대 서버로) Java API에 필요한 데이터 요청
3. Java API가 JSON 응답
4. Next.js 서버가 그 JSON으로 HTML을 렌더링
5. 완성된 HTML을 브라우저로 전송

## 상태 관리

프론트엔드의 "상태"는 그냥 브라우저 메모리에 떠 있는 JS 변수다. 백엔드/데이터베이스와는 전혀 상관없이, 지금 이 페이지가 열려있는 동안만 존재하는 임시 데이터다.

이 상태는 **새로고침하거나 탭을 닫으면 전부 사라진다.** 메모리에만 있던 값이기 때문이다. 그래서 "사라지면 안 되는 상태"는 별도로 어딘가에 저장해야 하는데, 그 선택지는:

- **localStorage / sessionStorage** — 브라우저 자체에 저장 (서버 없이, 그 브라우저에서만 유지). 예: 다크모드 설정
- **URL 쿼리스트링** — 예: `?page=2` — 뒤로가기/새로고침해도 유지되고 공유도 가능
- **백엔드로 전송해 DB에 저장** — 로그인 정보, 장바구니처럼 다른 기기에서도 봐야 하거나 영구히 남아야 하는 데이터

## React 핵심

컴포넌트 단위로 UI를 만드는 JavaScript 라이브러리다.

- **컴포넌트** — 화면을 재사용 가능한 조각으로 나눈 것. 함수 하나가 컴포넌트 하나.
- **JSX** — HTML처럼 생긴 JavaScript. JS 안에서 HTML 구조를 표현하기 위한 문법 설탕.

### 재렌더링 원리 ([Virtual DOM](/posts/virtual-dom/))

State가 바뀌면 React는:

1. 컴포넌트 함수를 다시 실행해 "이래야 할 화면"의 가상 표현(Virtual DOM)을 새로 만든다.
2. 이전 것과 비교(diffing)해서 실제로 달라진 부분만 진짜 화면(DOM)에 반영한다.

→ 전체를 다시 그리지 않고 바뀐 부분만 효율적으로 업데이트하는 게 React의 성능 비결이다.

### Hooks

함수 컴포넌트에 기능을 붙이는 방법이다.

- **useState** — 상태 하나 만들기
- **useEffect** — "화면이 그려진 후(또는 특정 값이 바뀐 후)에 실행할 일" 지정 → API 호출, 구독 설정 등에 씀
- **useContext** — 트리 전체에 값을 공유 (props 전달 생략)
- 이 외에도 `useMemo`, `useCallback`(성능 최적화), `useRef`(DOM 직접 접근) 등이 있지만 초반엔 `useState`/`useEffect`만 알아도 충분하다.

"상태(state)가 바뀌면, 그 상태를 쓰는 컴포넌트 함수가 재실행되고, 화면에서 바뀐 부분만 자동으로 업데이트된다" — 이 한 문장이 React의 전부라고 봐도 된다. CSR, API 요청, 상태 관리가 전부 이 원리 위에서 맞물려 돌아간다.

### 상태는 언제 바뀌나

상태는 저절로 바뀌지 않는다. 오직 set 함수(예: `setCount`)를 직접 호출할 때만 바뀌고, 그 호출은 항상 어떤 "계기(이벤트)"에 의해 일어난다. 대표적인 계기들:

**1. 사용자 행동 (가장 흔함)** — 클릭, 입력, 스크롤, 드래그 같은 인터랙션이 이벤트 핸들러를 실행시키고, 그 안에서 setState를 호출한다.

```jsx
<button onClick={() => setCount(count + 1)}>클릭</button>
<input onChange={(e) => setName(e.target.value)} />
```

**2. API 응답이 도착했을 때** — fetch가 끝나 데이터가 도착하는 시점이 계기다.

```jsx
useEffect(() => {
  fetch('/posts').then(res => res.json()).then(data => setPosts(data)) // 응답 도착 → 상태 변경
}, [])
```

**3. 시간이 흘렀을 때 (타이머)**

```jsx
useEffect(() => {
  const timer = setInterval(() => setSeconds(s => s + 1), 1000) // 1초마다 계기 발생
  return () => clearInterval(timer)
}, [])
```

**4. 다른 상태/props가 바뀌었을 때 (연쇄 반응)** — `useEffect`로 "이 값이 바뀌면 저 상태도 갱신"하는 식으로 연결할 수 있다.

```jsx
useEffect(() => {
  setIsValid(email.includes('@'))  // email이 바뀔 때마다 실행
}, [email])
```

**5. URL/라우트가 바뀌었을 때** — 페이지 이동도 라우팅 라이브러리 입장에선 하나의 이벤트라, 그에 따라 관련 컴포넌트의 상태가 갱신되곤 한다.

**중요한 함정: 변수를 직접 바꾸면 안 된다.**

```jsx
// ❌ 이렇게 하면 화면이 안 바뀜
count = count + 1

// ✅ 반드시 이렇게
setCount(count + 1)
```

`count = count + 1`처럼 변수를 직접 건드리면 값은 바뀌어도 React는 그 사실을 모른다. React는 오직 setState 함수 호출을 "감시"하고 있다가, 그게 호출되는 순간에만 "아, 다시 그려야겠다"고 반응한다. 그래서 "언제 상태가 바뀌냐"는 질문의 가장 정확한 답은 **"개발자가 setter 함수를 호출하는 그 순간"**이다. 그 함수를 호출하게 만드는 이벤트(클릭, API 응답, 타이머 등)는 다양하지만, 실제 변경은 항상 그 함수 호출을 통해서만 일어난다.

## Next.js

React 위에 얹어서 쓰는 [메타프레임워크](/posts/meta-framework/)다. (앞서 본 대로 자기만의 Node.js 서버를 갖고 SSR을 담당한다.)

## Node.js

JavaScript를 브라우저 밖에서도 실행할 수 있게 해주는 런타임 환경이다. 원래 자바스크립트는 브라우저 안에서만 동작했다. 브라우저 안에 엔진이 내장되어 있기 때문이다. Node.js는 이 엔진을 뜯어내서 컴퓨터에서 독립적으로 돌아갈 수 있게 만든 것이다.

## 타입스크립트를 쓰는 이유

자바스크립트에는 타입이 없다. 그래서 타입이 잘못되어도 실행할 때까지 모른다. 타입스크립트는 이런 오류를 **컴파일 시점에** 잡을 수 있게 해준다.

## Tailwind

유틸리티 클래스 방식의 CSS 프레임워크다. 미리 정의된 클래스를 조합해서 스타일링한다. 매번 컴포넌트의 색상·크기를 명시적으로 지정하는 대신 class를 사용하는데, 마치 `const` 변수 같은 느낌이다.

먼저 짚어둘 것: **CSS 클래스 자체는 Tailwind와 무관하게 원래 웹의 기본 기능이다.** Tailwind 없이도 얼마든지 클래스를 쓴다.

```html
<button class="save-button">저장</button>
```

```css
/* styles.css */
.save-button {
  background-color: blue;
  color: white;
  padding: 8px 16px;
}
```

이게 Tailwind 없이 쓰는 "일반 CSS"다. `class` 속성과 CSS 파일 문법은 HTML/CSS 표준 자체에 있는 기능이지, Tailwind가 발명한 게 아니다.

### 그럼 Tailwind가 다른 점은?

**"클래스 하나 = 스타일 하나(단일 속성)"** 로 미리 만들어둔 클래스들을 제공한다는 게 핵심이다. 이걸 유틸리티 클래스(utility class) 방식이라고 부른다.

```
┌───────────────┬──────────────────────────────┬─────────────────────────────────┐
│               │           일반 CSS           │            Tailwind             │
├───────────────┼──────────────────────────────┼─────────────────────────────────┤
│ 클래스 이름   │ 내가 직접 짓는다             │ 이미 정해져 있다 (bg-blue-500)  │
│               │ (.save-button)               │                                 │
├───────────────┼──────────────────────────────┼─────────────────────────────────┤
│ 클래스 하나의 │ 여러 속성 묶음               │ 딱 한 가지 속성만               │
│  역할         │ (배경+글자색+패딩 등)        │ (bg-blue-500은 배경색만)        │
├───────────────┼──────────────────────────────┼─────────────────────────────────┤
│ 스타일 정의   │ 별도 CSS 파일                │ HTML/JSX 태그 안에 클래스로     │
│ 위치          │                              │ 바로 조합                       │
└───────────────┴──────────────────────────────┴─────────────────────────────────┘
```

같은 버튼을 Tailwind로 만들면:

```html
<button class="bg-blue-500 text-white px-4 py-2 rounded">저장</button>
```

- `bg-blue-500` = 배경색만
- `text-white` = 글자색만
- `px-4` = 좌우 패딩만
- `py-2` = 상하 패딩만
- `rounded` = 모서리만

하나하나가 CSS 속성 하나씩에 대응하고, 이걸 레고 조립하듯 여러 개 붙여서 원하는 스타일을 만드는 방식이다. 반면 일반 CSS는 `.save-button` 클래스 하나 안에 이 모든 속성을 몰아넣는다.

### 그래서 실질적으로 뭐가 좋아지나

1. **이름 짓는 고민이 사라짐** — `.save-button-primary-lg` 같은 이름을 고민할 필요 없이, 원하는 스타일에 대응하는 기존 클래스를 조합만 하면 된다.
2. **CSS 파일과 HTML을 오갈 필요 없음** — 스타일이 바로 마크업 안에 있어서, 이 버튼이 어떻게 생겼는지 클래스만 봐도 바로 안다.
3. **일관된 디자인 값** — `px-4`처럼 미리 정해진 간격 체계(4px 단위 등)를 쓰게 강제되어, 팀원마다 `padding: 13px`, `padding: 15px`처럼 미묘하게 다르게 쓰는 걸 방지한다.
4. **안 쓰는 CSS 자동 제거** — 빌드 시 실제로 쓰인 클래스만 최종 CSS에 남기고 나머지는 제거해 파일 용량이 작아진다.

정리하면:

- **CSS class 자체** = 누구나 쓰는 표준 기능, Tailwind 없어도 100% 가능
- **Tailwind** = "미리 만들어둔 아주 잘게 쪼갠 클래스 모음 + 그걸 조합해 스타일링하는 방법론"을 제공하는 도구/철학일 뿐

## 패키지 매니저

속도와 디스크 사용 방식이 다르다.

1. **npm** — Node.js에 기본 내장. 가장 표준적. 예전에는 느렸으나 개선됨.
2. **yarn** — npm보다 빠르게 동작. 캐싱/병렬 처리로 속도 개선.
3. **pnpm** — 같은 라이브러리를 프로젝트마다 중복 저장하지 않고, 전역 저장소에서 [심볼릭 링크](/posts/symbolic-link/)로 공유한다. 디스크 공간 절약, 설치 속도 빠름. 신규 프로젝트에서 선호도가 높아지는 추세.

그래도 npm을 여전히 많이 쓰는 이유는? Node.js의 기본 도구인 데다, 이제 성능 격차가 크게 없고, 생태계 호환성이 좋기 때문이다. 작은 프로젝트에서는 셋의 차이가 크게 나지 않는다.

## 테스트

바람직한 테스트는 3단계 피라미드로 구성한다.

1. **단위 테스트** — 함수 하나, 컴포넌트 하나의 작동 방식 검증 (Vitest, Jest)
2. **통합 테스트** — 여러 컴포넌트 통합 테스트, API 연동 테스트
3. **E2E 테스트** — 실제 브라우저에서 사용자 흐름 전체를 재현 (Playwright)

API가 개발되기 전에는 [mock](/posts/mock/)으로 진행하는 식으로 먼저 완성한다.
