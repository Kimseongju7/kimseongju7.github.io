## 한 줄 정의
SSR(Server-Side Rendering, 서버 사이드 렌더링)은 사용자가 페이지를 요청할 때마다 **서버에서 완성된 HTML을 그려서 보내주는** 렌더링 방식이다.

## 자세히
웹 페이지의 화면(HTML)을 **누가, 언제 만드느냐**로 렌더링 방식이 갈린다. SSR은 방문자가 어떤 주소를 요청하면, 그 순간 서버가 필요한 데이터(DB 조회 등)를 채워 넣어 **이미 다 그려진 HTML**을 만들어 브라우저에 내려보낸다. 브라우저는 받은 HTML을 그대로 보여주기만 하면 되므로, 첫 화면이 빨리 뜨고 내용이 처음부터 HTML에 들어있어 검색엔진([SEO](/posts/seo/))이 읽기에도 좋다.

이걸 이해하려면 대비되는 두 방식과 같이 보는 게 좋다. **[CSR](/posts/csr/)(Client-Side Rendering, 클라이언트 사이드 렌더링)** 은 서버가 거의 빈 HTML과 자바스크립트 뭉치만 보내고, 브라우저에서 JS가 실행되며 화면을 그린다 — React·Vue 같은 [SPA](/posts/spa/)(Single Page Application)의 기본 방식이다. 첫 로딩은 느리고 SEO에 불리하지만, 이후 페이지 전환은 서버를 다시 거치지 않아 매끄럽다. **SSG(Static Site Generation, 정적 사이트 생성)** 는 요청 때가 아니라 **빌드 시점에 미리** 모든 HTML을 다 만들어 두는 방식이다([Jekyll·Chirpy](/posts/jekyll-chirpy/)가 이 방식). 가장 빠르고 서버가 필요 없지만, 내용이 바뀌면 다시 빌드해야 한다.

정리하면 **HTML을 언제 만드느냐**의 차이다 — SSG는 "미리(빌드 때)", SSR은 "요청받는 그때(서버에서)", CSR은 "브라우저 도착 후(클라이언트에서)". SSR은 SSG보다 최신 데이터를 반영하기 쉽고(요청마다 새로 그림), CSR보다 첫 화면·SEO가 유리한 중간 지점에 있다. 대신 요청마다 서버가 일을 하므로 서버 부하와 응답 지연이 CSR·SSG보다 커질 수 있다.

요즘 프레임워크는 이 방식들을 섞어 쓴다. 대표적으로 **Next.js**(React 기반)는 페이지·[컴포넌트](/posts/component/) 단위로 SSR·SSG·CSR을 골라 쓸 수 있고, 최근엔 서버에서 렌더링하는 **React Server Components(RSC)** 로 이 경계를 더 세분화한다. 즉 SSR은 "옛날 방식으로의 회귀"가 아니라, CSR의 SEO·초기 로딩 약점을 보완하려고 다시 주목받은 접근이다.

## 예시
같은 블로그 글 페이지를 세 방식으로 서빙한다고 하면:

- **SSG** — 글을 쓰고 빌드하면 `my-post.html`이 미리 생성됨. 방문자는 그 파일을 그냥 받음. (이 블로그가 쓰는 방식)
- **SSR** — 방문자가 접속하는 순간 서버가 DB에서 글을 꺼내 HTML을 그려서 응답. 댓글 수처럼 자주 바뀌는 값도 최신으로 반영 가능.
- **CSR** — 빈 껍데기 HTML + JS를 받고, 브라우저에서 JS가 API로 글 데이터를 가져와 화면을 그림.

Next.js에서의 대략적인 선택 (App Router 기준):

```jsx
// 서버에서 매 요청마다 렌더링 (SSR) — 항상 최신 데이터
async function Page() {
  const post = await fetch('.../post', { cache: 'no-store' }) // 요청마다 새로
  return <Article data={post} />
}

// 빌드 시점에 한 번 생성 (SSG) — 기본 캐시 동작
async function Page() {
  const post = await fetch('.../post') // 캐시되어 정적으로 생성
  return <Article data={post} />
}
```
