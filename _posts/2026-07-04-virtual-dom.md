## 한 줄 정의
가상 DOM(Virtual DOM)은 실제 [DOM](/posts/dom/) 트리를 본뜬 **가벼운 자바스크립트 객체 복제본**으로, 화면이 바뀔 때 이 복제본에서 먼저 변경을 계산하고 **실제로 달라진 부분만** 골라 진짜 DOM에 반영해 성능을 아끼는 기법이다.

## 자세히
실제 [DOM](/posts/dom/)을 직접 조작하는 것은 비싸다. 노드 하나를 바꿀 때마다 브라우저는 레이아웃을 다시 계산(reflow)하고 화면을 다시 그리는(repaint) 과정을 거친다. 화면이 조금만 바뀌어도 이 작업이 반복되면 앱이 느려진다. 특히 [컴포넌트](/posts/component/)가 상태(state)가 바뀔 때마다 "다시 그린다(re-render)"는 방식으로 동작하는 [React](/posts/react-doctor/) 같은 도구에서는, 매번 실제 DOM을 통째로 다시 그리면 감당이 안 된다.

가상 DOM은 이 문제를 **한 단계 완충**으로 푼다. 화면 상태가 바뀌면 React는 실제 DOM을 건드리는 대신, 메모리 안에 새로운 가상 DOM 트리(순수한 JS 객체)를 만든다. 그리고 바뀌기 전의 가상 DOM 트리와 새 트리를 **비교(diffing)**해서 "무엇이 실제로 달라졌는지"를 계산한다. 이 비교 과정을 **재조정(reconciliation)**이라 부른다. 마지막으로 달라진 최소한의 부분만 모아 실제 DOM에 **일괄 반영(batch)**한다. 즉 개발자는 "화면이 어떤 모습이어야 하는지"만 선언하면, 실제 DOM을 어떻게 최소한으로 고칠지는 가상 DOM이 알아서 처리한다.

핵심은 가상 DOM 자체가 빠른 마법이 아니라는 점이다. 가상 트리를 만들고 비교하는 데에도 비용이 든다. 가상 DOM의 진짜 가치는 **느린 실제 DOM 조작 횟수를 최소로 줄여준다는 것**과, 개발자가 DOM을 손으로 세밀하게 다루지 않고 **선언적으로(declarative)** UI를 짤 수 있게 해준다는 데 있다. "상태가 이러면 화면은 이렇다"만 쓰면 되고, 그 사이의 지저분한 DOM 갱신은 라이브러리가 대신 해준다. 그래서 가상 DOM은 성능 최적화 도구이자 동시에 **개발 편의성**을 위한 추상화다.

참고로 모든 프레임워크가 가상 DOM을 쓰는 것은 아니다. Svelte나 Solid 같은 최신 도구는 가상 DOM 없이 컴파일 단계에서 "어디를 바꿔야 하는지"를 미리 계산하는 방식을 택하기도 한다. 가상 DOM은 여러 해법 중 하나이지 유일한 정답은 아니다.

## 예시
```jsx
function Counter() {
  const [count, setCount] = React.useState(0);
  return (
    <div>
      <p>현재 값: {count}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}
```
버튼을 누르면 `count`가 바뀌고 컴포넌트가 다시 렌더링된다. 이때 React는:

1. 새 가상 DOM 트리를 만든다 (`현재 값: 1` 인 `<p>` 포함).
2. 이전 가상 DOM과 비교 → `<p>` 안의 텍스트만 `0 → 1` 로 달라졌음을 찾아낸다.
3. 실제 DOM에서는 그 텍스트 노드 하나만 갱신한다. `<div>`, `<button>` 은 건드리지 않는다.

화면 전체를 다시 그리는 대신 바뀐 숫자 한 글자만 실제 DOM에 반영되므로 빠르다.
